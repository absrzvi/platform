## UX Changes — Three Specific Updates

### 1. Fleet Manager — Triage Entry State (update to prior decision)

**Before:** Triage showed operational alert + vehicle status.

**After:** Add cost-impact one-liner per disruption.

**Why:** The Disponent im Eisenbahnwesen is explicitly responsible for cost management through fleet deployment. They make decisions that have direct cost signatures they own. Every AI recommendation must carry a cost implication at the moment of decision.

**Wireframe spec:**
```
Disruption card in triage:
  ┌────────────────────────────────────────┐
  │ 🔴 Unit 4722 — Door Motor Fault        │
  │    Car 4, Door 2L · Detected 06:14     │
  │                                        │
  │    Impact: 14-min delay, Route 12      │
  │    Cost: Replacing with Unit 4819      │
  │    adds ~€2,400 estimated maintenance  │
  │                                        │
  │    [View Detail]  [Assign Replacement] │
  └────────────────────────────────────────┘
```
- Cost line is one sentence, not a financial dashboard
- Cost data sourced from maintenance cost model (Rail MCP Server tool — add to tool signatures)
- "Estimated" qualifier always present — not false precision

### 2. Control Centre — Relay Card Component (new component, not in prior spec)

**Before:** Control Centre designed for generic landside operator.

**After:** Add relay card component specifically for Fahrdienstleiter (dispatcher) forwarding.

**Why:** Fahrdienstleiter is responsible for passenger announcements and conductor coordination. They need a ready-to-relay plain-language sentence, not an alert log or raw telemetry. This is a distinct UI component from the alert feed.

**Relay card spec:**
```
Relay Card component:
  ┌────────────────────────────────────────┐
  │ RELAY · Train 4712 · 14:32             │
  │                                        │
  │ "Train 4712 is delayed 14 minutes      │
  │  due to signal fault at Salzburg Hbf.  │
  │  Expected recovery: on departure."     │
  │                                        │
  │  [Copy]  [Forward to Conductor]        │
  └────────────────────────────────────────┘
```
- Generated by LLM from intent packet incident_summary
- Passenger-safe language (not technical fault codes)
- Copy = clipboard, Forward = pushes to conductor's PWA notification
- Lives in Control Centre alert rail alongside existing alerts
- Rendered distinctly from analysis-facing alert cards

### 3. Depot Briefing — ECM 1 Authorisation as Blocking State (update to prior decision)

**Before:** ECM Manager sign-off was one of several status items on vehicle card.

**After:** ECM 1 authorisation is an explicit blocking state, visually above technician sign-off in hierarchy.

**Why:** EU Directive requires ECM authorisation before vehicle returns to service. ECM Manager has regulatory blocking power. The visual hierarchy must reflect the regulatory hierarchy.

**Vehicle card state spec:**
```
Vehicle card states (in order):
  1. Technician faults open      → amber, technician action needed
  2. Technician faults resolved  → faults ACKed, awaiting ECM 1
  3. PENDING ECM 1 AUTHORISATION → distinct blocking state, blue/purple
                                   prominent, not just another tag
  4. ECM 1 AUTHORISED            → green, vehicle cleared to return
  5. DEFERRED                    → fault acknowledged, deferred to next
                                   maintenance window with reason

Blocking state renders:
  ┌────────────────────────────────────────┐
  │ ⏸ PENDING ECM 1 AUTHORISATION         │
  │   Unit 4722 · All faults resolved     │
  │   Awaiting: [ECM Manager name]         │
  │   Since: 06:42 (18 min)               │
  │                                        │
  │   [Review & Authorise]                 │
  └────────────────────────────────────────┘
```
- ECM Manager taps "Review & Authorise" → sees full fault resolution summary → signs off with deliberate hold gesture
- Blocking state cannot be bypassed by technicians
- Timer shows how long vehicle has been waiting for authorisation

## Landside Persona Accountability Model (Saga)

Each persona carries a specific kind of accountability. Design question for each interface: *what decision are they accountable for, and does our interface help them make it defensibly?*

| Persona | Accountability | Interface must enable |
|---|---|---|
| Dispatcher (Fahrdienstleiter) | Passenger communication + incident coordination | Plain-language relay, rapid crisis navigation |
| Fleet Coordinator (Disponent) | Cost management through fleet deployment | Cost-consequence-first data hierarchy, 48-72h planning horizon |
| ECM Manager (ECM 1) | Regulatory compliance — EU Directive liability | Defensible audit trail, blocking authorisation flow |
| Technicians (ECM 4) | Domain-specific fault resolution | Role-filtered depth, structured ACK record |

### Fleet Manager planning horizon
- Disponent makes 48–72h deployment decisions, not just real-time response
- Fleet Manager interface needs a forward planning view alongside real-time
- Triage state: present incidents + next 48h predicted issues
- Trend state: historical patterns for planning inputs

## Architecture Updates (Winston)

### ECM Sign-Off — Audit Trail Schema

```sql
-- Append-only. No UPDATE. No DELETE.
-- Row-level trigger raises exception on any modification attempt.
CREATE TABLE ecm_signoff_events (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  vehicle_id    TEXT NOT NULL,
  technician_id TEXT NOT NULL,          -- IdP-bound identity, not local session
  signed_off_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  vehicle_state JSONB NOT NULL,         -- snapshot of all faults at sign-off time
  directive_ref TEXT DEFAULT 'EU 2019/779',
  supersedes_id UUID REFERENCES ecm_signoff_events(id),  -- for corrections
  signature     TEXT                    -- HMAC for tamper-evidence
);
```

Rules:
- No UPDATE, no DELETE — ever
- Corrections = new row with `supersedes_id` pointing to corrected record
- `technician_id` must come from IdP (ÖBB SSO), not local user table
- `vehicle_state` JSONB = snapshot of all fault records at moment of sign-off
- Sign-off writes directly to landside PostgreSQL — NOT through edge event store
- Depot sign-off assumes landside connectivity (depot has fixed network)

### Shift Handover Schema

```sql
CREATE TABLE shift_handovers (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  outgoing_user   TEXT NOT NULL,
  incoming_user   TEXT NOT NULL,
  handed_over_at  TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  open_incidents  JSONB,               -- snapshot of live incidents at handover
  notes           TEXT                 -- freetext from outgoing Disponent
);
```

UI flow:
1. Outgoing Disponent taps "Hand Over Shift"
2. Sees summary of all open incidents
3. Optionally adds note
4. Confirms incoming Disponent (name/ID)
5. Record written to PostgreSQL
6. Incoming Disponent loads Fleet Manager → handover summary surfaced at top of triage view

Rail MCP Server: add `get_shift_handover(shift_id)` tool so Hermes can answer "what did the last shift leave me?"

### Operational State Rule
All operational state (active incidents, acknowledged alerts, assigned faults) must live in PostgreSQL — never in React component state or Hermes conversation memory. Every acknowledgement and status change is a server-side write. This is a hard architectural constraint.
