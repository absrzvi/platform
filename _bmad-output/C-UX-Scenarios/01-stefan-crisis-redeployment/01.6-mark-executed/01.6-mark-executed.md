## Overview

**Page Purpose:** Confirm Stefan's report that he executed the RAO externally (typically via HAFAS) and atomically write the shadow audit row capturing the recommendation → action → outcome chain. Uses the **same hold-to-record gesture** Klaus uses in 03.6 — the trust-building muscle memory locked across MVP and Phase 2.

**User Situation:** Stefan has read the RAO (01.5), opened HAFAS in another tab, executed the rotation update himself, confirmed by radio with the conductor. He returns to the platform to log that he did it.

**Success Criteria:**
- Confirmation surface explicitly labels Stefan as the executor and the shadow row as a *companion record* (not authoritative dispatch)
- 3s hold → 5s countdown → atomic write gesture identical to 03.6
- Idempotency key generated at countdown initiation
- Background HAFAS read poll launches at write completion
- Stefan receives feedback within 1s of countdown expiry

**Entry Points:**
- "Mark executed" button on RAO card (01.5)

**Exit Points:**
- Successful write → [01.7-execution-log](../01.7-execution-log/01.7-execution-log.md) (returns to triage with RAO in execution log)
- Cancellation during countdown → returns to [01.5-rao-card](../01.5-rao-card/01.5-rao-card.md), no write
- Write failure → error variant (toast + retry option); RAO not yet marked executed

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Default | Modal opened from RAO card | All sections visible; route default = HAFAS; hold button idle | Toggle route, type notes, hold, cancel |
| Hold initiating | User presses hold button | Button border progresses left-to-right over 3s | Release to cancel |
| Countdown | After 3s hold | Ring around button + countdown number 5→4→3→2→1; modal disables other interactions | Release or Cancel button → cancel |
| Writing | At countdown expiry | Button flashes; "Recording…" briefly visible | None |
| Success | Write succeeded | Modal closes; toast appears: "Recorded — RAO marked executed at {timestamp}"; redirect to 01.7 | Auto-redirect |
| Idempotency conflict (409) | Server returned 409 with same idempotency key | Treated as success — no error; same redirect | Auto-redirect |
| Network error | Write failed (5xx, timeout) | Toast: "Couldn't record — retry?"; modal remains open; hold button resets to idle | Retry, Cancel |

## Technical Notes

- **Write target:** `POST /api/audit/shadow-events` with payload `{ rao_id, action_route, action_route_detail?, notes?, idempotency_key, hold_started_at, countdown_completed_at, atomic_write_requested_at }`. Server enforces append-only on `ecm_signoff_events` and the broader audit table.
- **Idempotency:** key generated client-side at countdown initiation (`crypto.randomUUID()`); same key replays return 409 with prior result body, treated as success.
- **Background HAFAS poll:** at successful write, server queues a job to poll HAFAS read endpoint for the relevant rotation; outcome verification updates the row asynchronously. UI in 01.7 reflects the eventual outcome status.
- **Motion-reduced variant:** when `prefers-reduced-motion: reduce`, the radial ring animation is replaced with a simple count of seconds in text; the gesture still requires 3s hold + 5s countdown.
- **Verb rule reminder:** the button text is "Hold to record" — the *operator* records, the *platform* simply persists. NEVER "Record sign-off automatically" or "System recording…".
