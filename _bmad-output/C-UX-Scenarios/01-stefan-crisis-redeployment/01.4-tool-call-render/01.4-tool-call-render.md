## Overview

**Page Purpose:** Inline rendering of a Rail MCP tool call within the chat history — shows the tool name, parameters, result, and Kondukt's natural-language interpretation. This is *not* a separate page; it is the canonical chat-message component for tool-call turns. Documented separately because tool-call rendering is a defining grammar element that recurs throughout the platform.

**User Situation:** Stefan asked Kondukt a question in 01.3 ("What does deploying 4819 cost vs keeping 4722?"). Kondukt is calling `get_maintenance_cost_estimate`. Stefan needs to see both the structured tool output AND Kondukt's interpretation, with the option to expand or collapse the structured detail.

**Success Criteria:**
- Tool name + parameters visible at-a-glance (pill at top of message)
- Tool result expandable; collapsed by default for low-confidence or by user pref; expanded by default for high-confidence and structured-data tools
- Tool latency surfaced (response time + Rail MCP tool 8s hard timeout indicator per PRD)
- Confidence pill on the response

**Entry Points:**
- Inline in 01.3-crisis-fullwidth chat history (as a streamed Kondukt response containing a tool call)

**Exit Points:**
- Stefan asks follow-up → another chat turn (same page)
- Stefan asks "What does it take to redeploy?" → Kondukt returns an RAO → [01.5-rao-card](../01.5-rao-card/01.5-rao-card.md)
- Tool fails → error variant within same message

## Layout Structure

A single chat message bubble (Kondukt-authored) containing four stacked regions: tool-call pill (header), result viewer (collapsible), natural-language interpretation, confidence pill (footer).

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Streaming | Kondukt response in progress | Tool pill renders immediately; result + interpretation stream in; confidence pill appears last | None — wait |
| Complete | Stream finished | All sections rendered; chevron interactive | Expand/collapse, copy, follow-up |
| Tool error — recoverable | MCP tool returned `{error, recoverable: true}` | Result viewer shows error structure (red border); interpretation says "Retrying..."; auto-retry once per PRD | Watch |
| Tool error — non-recoverable | MCP tool returned `{error, recoverable: false}` or timed out at 8s | Result viewer shows error structure; interpretation says "Data source unavailable for [tool name]"; no retry; suggest fallback ("Try direct tool call?") | Direct tool, retry manually |
| Long output | Result JSON >40 lines | Result viewer collapsed by default with "Show full result (N lines)" affordance | Expand |

## Technical Notes

- **Tool call stream contract:** SSE events of types `tool_call_started`, `tool_call_param_delta`, `tool_call_result`, `tool_call_error`, `interpretation_delta`, `confidence_set`. Client renders each as it arrives.
- **8-second hard timeout:** per PRD. Client surfaces "Data source unavailable" with `{error, detail, recoverable}` structure. Recoverable errors retry once.
- **Tool names are NOT prettified:** raw `get_maintenance_cost_estimate` shown, not "Get Maintenance Cost Estimate." Reasons: (a) unambiguous mapping to backend logs for debugging; (b) signals "this is a real tool call, not magic"; (c) Stefan learns the tool surface implicitly.
- **Result viewer copy affordance:** small "Copy JSON" button on hover (Stefan persona F2 — reconstructability gap — make tool result easy to paste into incident notes).
- **shiki theme:** match overall light theme; respect prefers-color-scheme.
