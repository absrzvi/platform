## Overview

**Page Purpose:** Render a Recommended Action Object (RAO) as a readable card with everything Stefan needs to make a judgment — the structured action payload, justification, step-by-step execution procedure, confidence pill, and operator-action buttons ("Mark executed", "Modify"). **This is the defining component of the platform.** Every grammar choice in this spec sets precedent for every other RAO surface (scenarios 02, 06, plus all future Phase 2 dispatch surfaces).

**User Situation:** Stefan asked Kondukt "What does it take to redeploy 4819?" Kondukt returns an RAO. Stefan reads, judges, then either marks executed (accept) or modifies (scenario 02). In scenario 01, he accepts.

**Success Criteria:**
- Stefan reads payload + justification + step-by-step + confidence in one scan
- "Mark executed" is the *only* execution-shaped affordance; no "Execute," "Apply," "Confirm" verbs anywhere
- Step-by-step procedure pre-stages the exact HAFAS payload Stefan needs to enter
- Confidence pill is visible on the card surface, not behind expand

**Entry Points:**
- Stefan's free-form query in chat returns a structured RAO response (subtype of chat message)

**Exit Points:**
- "Mark executed" → [01.6-mark-executed](../01.6-mark-executed/01.6-mark-executed.md) confirmation
- "Modify" → modification panel ([02.4-modify-panel](../../02-stefan-cost-aware-triage/02.4-modify-panel/02.4-modify-panel.md))
- Stefan ignores and continues asking questions → another chat turn (01.3)

## Layout Structure

A specialised chat-message variant rendered as a *bordered card* (distinct from regular chat messages). Five stacked regions: header (RAO badge + confidence pill), action payload (structured), justification (prose), step-by-step procedure, operator-action footer.

```
┌── RAO Card ──────────────────────────────────────────────┐
│  ▢ RAO              · 2.4s        [CONFIDENCE: MEDIUM]  │
│  ───────────────────────────────────────────────────────  │
│                                                            │
│  ACTION                                                   │
│  ┌─────────────────────────────────────────────────────┐  │
│  │ type:       FLEET_REDEPLOYMENT                       │  │
│  │ system:     HAFAS                                    │  │
│  │ method:     rotation_update                          │  │
│  │ payload:                                             │  │
│  │   rotation_id:     22:15-WIEN-SBG                    │  │
│  │   effective_at:    22:15 today                       │  │
│  │   replacement_unit: 4819                             │  │
│  └─────────────────────────────────────────────────────┘  │
│                                                            │
│  WHY                                                      │
│  Unit 4722 cleared by ECM at 06:08. Unit 4819 is the      │
│  next available replacement. Cost: ~€2,400 maintenance    │
│  overhead. Confidence: medium — 4819 last serviced 34     │
│  days ago, approaching mileage threshold.                 │
│                                                            │
│  STEPS                                                    │
│  1. Open HAFAS                                            │
│  2. Navigate to Rotation 22:15                            │
│  3. Replace 4722 with 4819                                │
│  4. Confirm with conductor by radio                       │
│                                                            │
│  ───────────────────────────────────────────────────────  │
│  [ Modify ]              [ Mark executed ]                │
└──────────────────────────────────────────────────────────┘
```

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Default | RAO rendered after Stefan's question | All sections visible; both action buttons enabled | Modify, Mark executed |
| Streaming in | Mid-render | Sections appear top-to-bottom as stream arrives; action buttons disabled until confidence pill present | Wait |
| Marked executed | After 01.6 confirmation | Card transitions to read-only state; "Mark executed" button replaced with badge "EXECUTED · {timestamp} · Stefan Weber"; "Modify" no longer available | None — card is now historical |
| Modified + executed | After 02.4 + 01.6 flow | Card shows the modified payload (original visible via toggle/expand); badge "EXECUTED WITH MODIFICATION · {timestamp} · Stefan Weber"; reason text visible | Expand original |
| Low confidence | confidence == "low" | Confidence pill in outline style (dashed 1.5px border, "LOW" small-caps tracking-wide). Card left-rail desaturates to neutral-400. ## Technical Notes

- **RAO schema (locked):** `{ rao_id, generated_at, action: {type, system, method, payload}, justification: string, steps: string[], confidence: "high"|"medium"|"low", confidence_rationale: string, source_tools: string[] }`. Backend uses structured-output enforcement (Anthropic / OpenAI structured outputs feature) to guarantee the shape — not regex-parsing.
- **Source tools field:** lists which Rail MCP tools fed into this RAO. Surfaced in audit export. Not shown on card surface in MVP, but persisted.
- **Verb rule enforcement at button level:** the button text is hardcoded; not a translatable string that downstream localisation could weaken. EN translation key exists but the rule supersedes it.
- **Low-confidence treatment:** the extra confirmation step (state row above) is the same Hold-to-record gesture used in 01.6 — but Stefan must complete it twice (the second one is the extra friction for low-confidence). Phase 4 open question: confirm with Roland.
- **"Mark executed" is *not* the platform writing to HAFAS.** It is Stefan reporting that *he* executed via HAFAS. Background process polls HAFAS read endpoint to verify state change (see 01.6).
- **Chat-message rendering:** RAO card is a sub-type of chat message; in the chat virtualised list, it occupies more height (typically 320–400px) than a regular message bubble.
