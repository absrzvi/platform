## Overview

**Page Purpose:** The macro situational awareness surface for Lukas's three-zone Telematics Control Board. Zone 1 (top-left, dominant — ~55% of viewport) presents the **fleet compatibility matrix** with a synchronised geospatial telemetry overlay. From here Lukas filters across fleet × commissioning-stage × alert-state, identifies which units share a fault pattern, and selects one to drive Zone 2 (topology) and Zone 3 (Kondukt RAO console).

**User Situation:** Lukas is at his programme management workstation, dual-screen. He has Stadler Service Request #SR-2026-0517 open in another tab — Stadler blames ÖBB's landside reservation database for reservation-screen blackouts on three Cityjet DOSTO Neu units. He needs to identify the three units and inspect their cross-domain telemetry before he can build a defensible counter.

**Success Criteria:**
- Lukas can filter the matrix on `fleet × stage × alert-window` in ≤2 interactions and land on the three affected units within 10 seconds of page load
- The geospatial overlay stays in lockstep with the matrix filter (selecting a row highlights the unit's last-known position on the map; selecting a map marker highlights the row)
- Severity-banded alert state is visible at the **unit-row level** without drill-down — same glanceability contract as Stefan's triage cards
- Zone 1 selection drives Zone 2 binding in <1s (perceived) — see [06.2](../06.2-zone-1-to-2-binding/06.2-zone-1-to-2-binding.md)

**Entry Points:**
- Default page after IdP-authenticated login when the operator's role claim includes `telematik`
- Direct URL: `/telematics/control-board` (TanStack Router route)
- Return target after every other Telematics Control Board page (RAO log close, warranty packet PDF download, topology drawer close)

**Exit Points:**
- Lukas selects a unit row → Zone 2 topology binds → [06.2-zone-1-to-2-binding](../06.2-zone-1-to-2-binding/06.2-zone-1-to-2-binding.md)
- Lukas opens Zone 3 chat directly (no unit selected) → free-form Kondukt session (out of scenario 06's path; covered in 07)
- Lukas clicks "OTA rollout view" in chrome → scenario 07 entry
- Lukas signs out → IdP sign-out flow

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Default (no selection) | Lukas authenticated; filters at default; no row selected | Zone 1 active (matrix + map populated), Zone 2 idle, Zone 3 idle | Filter change, row click, map marker click, chat input click |
| Loading (initial) | First page render <2s | Skeleton matrix (8 placeholder rows) + map shimmer; Zone 2/3 idle as normal | None until loaded |
| Loading (filter change) | Filter re-query in flight | Matrix rows dim to 60% opacity; small spinner in filter bar; map markers fade | Filter clicks queue; no row clicks |
| Empty matrix | Filter combination returns 0 rows | Centered empty state in matrix area: "No units match these filters. Adjust filters or pick {default-fleet}." Map shows empty ÖBB network. | Filter changes only |
| Stale data (per row) | IntentPacket gap >5 min on a specific unit | Affected row: small staleness badge after last-alert column ("Stale — last update {time}") | Row click still works; staleness surfaced in topology drawer too |
| Vendor portal degraded | ≥1 vendor portal unreachable | Footer count visually degrades; row-level vendor-portal pill shows `confidence-low` outline for affected vendors | Cmd+click on row's vendor-portal pill still attempts the deep-link; failure surfaced to user |
| Kondukt degraded | Sustained Kondukt failure | Zone 3 strip transitions to degraded variant (greyed status, chat input replaced with `degraded-banner`: "Kondukt unavailable — matrix + topology remain functional. Vendor portal deep-links still work."); Zones 1 + 2 unaffected | Filter, row click, map click, vendor portal cmd+click |
| HAFAS degraded | HAFAS poll failed | Footer status shows degraded; position labels in matrix show "Position unavailable — last known {time}" inline | All other actions still work |

- **Performance contract:** Initial render of matrix + geospatial overlay < 2s under PoC fleet size (10–20 vehicles). PRD FR generalises from Stefan's <3s to Lukas's <2s here because the matrix is denser data per row and slower-feeling lag erodes Lukas's "scan-and-find" mental model.
- **Streaming:** Matrix rows update via SSE from `/api/telematics/fleet-matrix/stream` (downstream of MQTT IntentPacket ingest). Idempotent upsert on `(unit_id, alert_window)`.
- **Map tile source:** MapLibre GL JS with vector tiles served from a self-hosted tileserver (OSM-based + ÖBB rail network overlay). No third-party tile provider — sovereignty constraint from PRD.
- **Vendor portal health polling:** `/api/telematics/vendor-portals/health` every 60s. Cached client-side; degraded after 3 consecutive failures.
- **Virtualisation:** `@tanstack/react-virtual` on matrix table — supports ≥100 rows without scroll jank. Map markers also virtualised via MapLibre's native clustering at low zoom levels.
- **Sort order:** Severity-banded, then last-alert-newest within band, then alphabetical unit ID as deterministic tiebreaker. Same banded-hybrid grammar as scenario 01.1 triage feed — single shared sort utility.
- **Row → Zone 2 binding:** Selection state is held in the page's TanStack Router URL search-param (`?unit=DN-014`). Refresh-safe; deep-linkable. See [06.2](../06.2-zone-1-to-2-binding/06.2-zone-1-to-2-binding.md) for the transition spec.
- **Keyboard navigation:** Arrow keys cycle matrix rows; Enter selects. Cmd+F focuses the alert-type filter. Cmd+K opens chat input. Tab order: chrome → filters → matrix → map → Zone 3.
