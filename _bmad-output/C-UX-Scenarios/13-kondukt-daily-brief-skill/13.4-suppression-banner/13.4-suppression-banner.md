## Overview

**Page Purpose:** When the constitution-as-contract validator rejects a skill emission — schema violation, citation-trace failure, advisory-language breach, or tool-timeout citation-trace failure — the emission is **suppressed**. No brief reaches Roland. Instead, a suppression banner appears in the canvas with the reason. The recommendation triplet is still written to the archive (Envelope partially populated + Provenance Bundle from MCP calls + Validation Receipt with disposition=suppressed + suppression_reason). Renate can inspect suppressions in her audit surface (scenario 05.8 Suppression log inspection).

This is the FR25 surface — *silence is safe*. No regeneration loop in MVP. Roland's options: re-invoke the skill manually, simplify the query (for free-form), or wait for the underlying condition to resolve (e.g. tool timeout — wait for the MCP service to recover).

This sub-page is critical for Renate's Phase 2 trust criterion 3 (Provenance Replay archive integrity — including all the recommendations the validator *rejected*, not just the ones it emitted). The auditor's question at manday 20–25: "show me what the harness rejected and why" → Roland's suppression banners (Roland-facing surface) + Renate's Suppression log (audit-facing surface) are the two sides of the same archive entry.

**User Situation:** Roland invoked the *Daily Brief* skill (13.1). Skill ran for ~15s. The LLM produced a draft. Validator failed: the LLM cited an IntentPacket ID that doesn't exist in the archive (citation-trace check failure). Emission suppressed. Roland now sees a banner in the canvas instead of the brief.

**Success Criteria:**
- Suppression banner clearly visible — informs Roland that *no recommendation reached him* and *why*
- Reason text plain-language: "Citation-trace check failed — the LLM cited an IntentPacket that doesn't exist in the archive."
- Suppression recorded in archive with disposition + suppression_reason fields (FR29 Validation Receipt)
- Roland's options visible: re-invoke skill, switch skill, navigate elsewhere
- No silent failure — operator is *informed* that a suppression happened (not just empty canvas)
- LOW-rate trip-wire monitoring continues (the suppression rate is the operator-facing metric; >5% sustained = calibration review)

**Entry Points:**
- From 13.1 after skill run completes with validator failing
- Browser refresh / direct URL on a suppressed session (rare; suppression banner persists for the session)

**Exit Points:**
- Click "Re-run Daily Brief" → back to [13.1](../13.1-skill-invocation/13.1-skill-invocation.md) (new invocation, new triplet)
- Click "Switch to Live Board" / "Switch to Kanban" → exits Kondukt view
- Click "Open suppression detail in archive" → opens Replay archive surface with this suppressed triplet's detail (scenario 05.8 — Renate's surface, but Roland can read his own suppressions)
- Click another skill button → invokes different skill
- Type in chat input → free-form chat (scenario 14)

## Components

### Skill Registry Bar (suppression indicator)

| Element | Behaviour |
|---|---|
| Daily Brief button | After suppression: shows small ⚠ indicator overlay on the skill icon for 30 seconds; tooltip: "Most recent invocation suppressed — see banner below"; click re-invokes |
| Visual state | Button remains enabled (suppression is not a disable condition); user can re-invoke immediately |

### Suppression Banner (main content)

Card-style banner occupying centre of canvas. Components:

#### Header

| Element | Content |
|---|---|
| Title | "⚠ Kondukt suppressed a recommendation" — warning iconography, not error (this is the *correct* harness behaviour) |
| Skill identifier | "Skill: Daily Brief v1.0" |
| Triplet ID | "Triplet: rcmnd_2026_05_25_06_47_a7c… (suppressed)" — clickable to open in archive |

#### Reason Block

| Element | Content |
|---|---|
| Reason category | Plain-language summary — e.g. "citation-trace check failed", "schema violation", "advisory-language breach", "tool-timeout: get_recurring_failure_pattern" |
| Reason detail | 2–3 sentence operator-friendly explanation. Example: "The LLM's draft cited IntentPacket pkt_05-25_02-14_DN-099 which does not exist in the archive. The validator rejected the emission rather than letting an unsupported claim reach you." |

Reason categories (matching `suppression_reason` enum values in FR25):

- `schema_violation` — LLM output didn't match the skill's output schema (missing fields, wrong types, etc.)
- `unsupported_citation` (citation-trace failure) — LLM cited an IntentPacket / MCP response that doesn't exist or doesn't support the claim
- `advisory_language_breach` — LLM used imperative-action language ("dispatch X") instead of advisory-only ("recommend dispatching X")
- `tool_timeout` — an MCP tool call timed out, so the LLM couldn't ground claims in fresh data; validator suppresses to avoid stale-data emission
- `provider_429` — BYOK provider returned 429 (rate limit); covered more broadly in scenario 10 harness-degraded, but a one-off 429 surfaces here
- `model_output_malformed` — LLM output couldn't be parsed as structured data; defensive last-resort check

#### Validation Receipt Display

| Element | Content |
|---|---|
| 3-check display | ✓ schema · ⚠ citation-trace (1 of 4 failed) · ✓ advisory-language |
| Click on ⚠ | Expands inline detail: which citation failed, what the LLM cited, what was checked against, why the check failed |
| Click on ✓ | Same as 13.2 — validator-rule popover for transparency |

#### Operator Options Section

| Element | Behaviour |
|---|---|
| "Re-run Daily Brief" | Primary CTA — most common operator response. New invocation; new triplet. The previous suppressed triplet remains immutable in archive. |
| "Open suppression detail in archive" | Secondary — opens Replay archive scenario 05.8 surface with this suppressed triplet detailed |
| "Switch to Live Board" / "Switch to Kanban" | Tertiary — exits Kondukt view without resolution |

#### Footer: "About suppression"

| Element | Content |
|---|---|
| Section header | "ⓘ About suppression" |
| Explanatory text | "The harness validates every recommendation before it reaches you. When validation fails, the recommendation is rejected — not regenerated. 'Silence is safe' — you never see a hallucinated claim." |
| Link | "[Learn more about validator rules]" — opens skill inspection panel (FR22) — same surface as 13.1 skill-inspection-on-info-icon |

This footer is **always present** on suppression banners. Operators (especially during early pilot mandays) will be unfamiliar with suppression as a UX concept; the explanatory framing reduces anxiety ("did the system break?" → no, "this is how the harness keeps you safe from hallucinations").

### Status Strip (with suppression count)

The status strip adds an operational metric:

| Element | Content |
|---|---|
| Suppression count today | "1 suppression today" — counts validator-rejected emissions in current 24h window |
| Click | Opens suppression log inline (operator-facing; lightweight — full audit view is Renate's 05.8 surface) |

This is the LOW-rate / suppression-rate trip-wire signal at operator-visible level (per FR63 calibration metric — sustained >5% triggers review).

### Recommendation Triplet (in archive — suppressed)

Even though no brief reaches Roland, the archive entry is fully populated:

| Component | Content |
|---|---|
| Envelope | `{recommendation_id, train_id (null for fleet-wide brief), recommended_action (null), advisory_class: "daily_brief", confidence: null, timestamp, harness_version, model_provider, model_version, skill_invoked: "daily_brief", operator_id, justification: null}` |
| Provenance Bundle | `{intent_packets_consumed: [list from MCP calls], mcp_queries: [...with response_hashes], allowlisted_fetches: [], llm_raw_output: full LLM draft string}` |
| Validation Receipt | `{schema_check: pass, citation_trace_check: fail, citation_trace_failure_detail: {claimed_packet_id: "pkt_05-25_02-14_DN-099", reason: "not_found_in_archive"}, advisory_language_check: pass, modification_distance: null, disposition: suppressed, suppression_reason: "unsupported_citation"}` |

The full Provenance Bundle is preserved — this is what makes the Replay CLI re-verification possible six months later. Renate's auditor can pull this triplet from archive, run the validator deterministically, and confirm: "yes, this was correctly suppressed."

## State Variants

| State | Trigger | Visual |
|---|---|---|
| Default (suppression banner displayed) | Skill invoked → validator failed | Full banner as above; Re-run CTA primary |
| Suppression on first invocation in shift | Roland's first Daily Brief attempt suppressed | Same banner; no historical context; Re-run + Switch options |
| Repeated suppression (same reason) | Roland re-invoked + validator failed again with same reason | Banner shows additional context: "This is the {N}th consecutive suppression with reason {reason}. Consider: try later (if tool-timeout) / contact Nomad ops (if persistent)" |
| Repeated suppression (different reason) | Roland re-invoked + validator failed with different reason | Standard banner for the new reason; subtle indicator of previous suppression: "{N} previous suppressions in this session" |
| Tool-timeout suppression (BYOK provider 429) | Provider returned 429 during skill run | Same banner with reason category `provider_429` (or `tool_timeout` if MCP); additional message: "Your LLM provider returned 429 (rate limit). Token usage today: {n}. Consider checking your provider quota." |
| LLM injection detection suppression | Validator's advisory-language linter detected prompt-injection-style output | Same banner with reason `advisory_language_breach` (or specific injection-detection sub-reason); **FR57 security event emitted to ÖBB IT Security pipeline**; subtle indicator: "This event was reported to ÖBB IT Security per platform safety policy" |
| Suppression with multiple validator failures | Validator rejected on >1 check | Validation Receipt shows multiple ⚠ marks; reason block lists primary failure (most critical), notes secondary failures inline |
| Background re-invocation (Phase 1.5; not MVP) | Operator could configure auto-retry — MVP working assumption: not available | N/A in MVP — operator manually re-invokes |

## Behaviour Notes

- **No regeneration in MVP.** Per FR25 — the LLM is not given a second chance with feedback. Roland's path is manual re-invocation. Phase 1.5 may add bounded retry-with-error-feedback.
- **Suppression IS a recordable outcome.** The triplet (Envelope + Provenance Bundle + Validation Receipt with disposition=suppressed) is written to archive identically to emitted triplets. This is the audit-trail completeness guarantee — Renate's auditor sees both what the harness emitted AND what the harness rejected.
- **Operator interaction logged per FR32:**
  - `kondukt_suppression_banner_displayed` — when banner first renders for a suppression (engagement signal)
  - `kondukt_suppression_banner_read_ms` — dwell-time on banner (telemetry — did Roland read the reason or skim?)
  - `kondukt_re_invoke_after_suppression` — when Roland clicks Re-run after suppression (key signal for the modification-rate / acceptance-rate ratio)
  - `kondukt_suppression_archive_opened` — when Roland clicks "Open suppression detail in archive" (Phase 2 trust signal — operators self-auditing)
- **Suppression rate tracking:**
  - **Per-skill** — Daily Brief suppression rate vs Vendor meeting prep suppression rate
  - **Per-reason** — citation-trace failures vs schema violations vs tool-timeouts
  - **Per-day rolling window** — feeds the seven-criteria Phase 2 composite (criterion 3 Replay archive integrity)
  - **Per-validator-rule** — which validator rule rejects most often (Phase 4 / Phase 1.5 ops view)
- **Repeated suppression handling.** If Roland re-invokes 3 times consecutively and all suppress with the same reason, the banner adds a "consider escalating" message. This is a friction-reducing affordance, not a workflow change — Roland still owns the decision.
- **Performance:** suppression banner render <200ms after validator rejection. No blocking spinner; canvas transitions from running state to suppression banner state directly.
- **Accessibility:** banner is announced to screen-reader on render — "Kondukt suppressed a recommendation. Reason: citation-trace check failed. {full reason text}. Options available below."

## Open Questions

- **Reason taxonomy completeness** — 6 categories listed (`schema_violation`, `unsupported_citation`, `advisory_language_breach`, `tool_timeout`, `provider_429`, `model_output_malformed`). May need a 7th for "validator timeout" (validator itself slow / hung) or "constitution version mismatch" (skill was invoked but constitution had been updated mid-run). Phase 4 confirms enum completeness with engineering review.
- **Plain-language reason text — per-skill or per-reason** — currently spec'd as per-reason (same text regardless of skill). Some operators may benefit from skill-specific framing (Daily Brief vs Vendor meeting prep). Working assumption: per-reason for MVP; per-skill-and-reason for Phase 1.5 if friction reported.
- **Repeated suppression threshold** — banner adds "consider escalating" after N consecutive suppressions. N=3 working assumption. Too low = noise; too high = late warning. Phase 4 confirms with pilot data.
- **Background re-invocation** — Phase 1.5 may add auto-retry-with-backoff for tool-timeout suppressions specifically (these are often transient). MVP requires manual re-invocation. Phase 4 confirms whether tool-timeout warrants special-case auto-retry.
- **Operator-side suppression log surface** — currently spec'd as "click suppression count → opens inline log". This is lightweight (counts + timestamps + reasons); the audit-facing detail (full Provenance Bundle inspection) is Renate's 05.8 surface. Phase 4 confirms whether Roland needs more depth on his own surface.
- **LLM injection detection threshold** — what triggers the FR57 security event vs a normal `advisory_language_breach` suppression? Working assumption: signature-based detection (specific injection patterns) triggers security event; vanilla language breach (e.g. accidental "dispatch X" phrasing) does not. Phase 4 confirms with security team review.
- **"About suppression" footer placement / persistence** — always present on banner working assumption. Operator may grow tired of seeing it after weeks of pilot use. Phase 1.5 may add per-user dismiss / suppress-help-text preference. MVP: always show.
- **Banner dismissal** — Roland can navigate away to Live Board / Kanban; banner is replaced by those views. On return to Kondukt, does the suppression banner persist? Working assumption: yes (suppression is a session-relevant event); persists until Roland either re-runs the skill or explicitly dismisses (× affordance — Phase 4 may add).
