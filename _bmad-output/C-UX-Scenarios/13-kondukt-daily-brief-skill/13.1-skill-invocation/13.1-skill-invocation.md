## Overview

**Page Purpose:** The Kondukt view at idle — chat input + skill registry shortcut bar visible; canvas empty (no prior session content). Roland invokes the *Daily Brief* skill via button click, keyboard shortcut (Cmd/Ctrl+1 mapped to Daily Brief by default), or chat command `/daily-brief`. The skill invocation event triggers the harness to:

1. Construct the skill's MCP tool call chain (`get_fleet_telemetry` + `get_recurring_failure_pattern` + `search_fault_history`)
2. Dispatch the prompt to the customer's BYOK LLM
3. Receive LLM draft output
4. Validate through schema + provenance + advisory-language linter
5. Emit (→ 13.2) OR suppress (→ 13.4)

This sub-page documents the invocation surface itself — the moment Roland's intent becomes a skill run.

**User Situation:** Roland clicked "Kondukt" in top chrome. View loads. Empty canvas. Skill registry shortcut bar visible at top of view. Chat input below.

**Success Criteria:**
- Skill registry shortcut bar visible with four MVP skills (Daily Brief, Recurring-failure pattern, Vendor meeting prep, Fleet health summary)
- Click on skill button initiates the skill run
- Status indicator shows skill running state with elapsed time
- Skill registry inspectability — Roland can click a skill name's info icon to see constitution + schema + validator rules (FR22)
- Chat input as alternative — `/daily-brief` typed and Enter pressed has equivalent effect
- Keyboard shortcut — Cmd/Ctrl+1 invokes Daily Brief (configurable per-user)

**Entry Points:**
- Top-nav click from Live Board / Kanban
- Browser direct URL `/kondukt`
- Keyboard shortcut Cmd/Ctrl+K (opens Kondukt from anywhere in harness)
- Deep-link from another view (e.g. "Open in Kondukt" affordance from a Kanban card — Phase 4 may add)

**Exit Points:**
- Click *Daily Brief* skill button → invocation triggered → skill running state → next state per outcome (13.2 or 13.4)
- Click another skill button → that skill's flow (scenario 14 for Vendor meeting prep; other skills out of MVP scope)
- Type chat message in input + Enter → free-form chat flow (out of this sub-page's scope; handled in scenario 14's chat sub-page)
- Navigate to Live Board / Kanban → exits Kondukt view

## Components

### Skill Registry Shortcut Bar

| Element | Behaviour |
|---|---|
| Skill button | Card-style button per skill. Default state: skill icon + name + keyboard shortcut hint + info icon (ⓘ). Hover: subtle elevation. |
| Skill button click | Invokes the skill. Click animation; status indicator transitions to "running" state. |
| Info icon (ⓘ) click | Opens skill inspection panel — shows constitution excerpt + output schema + provenance rules + advisory-language linter constraints + skill version (per FR22). Inline expansion or modal — Phase 4 confirms. |
| Keyboard shortcut hint | Visible on button — Cmd/Ctrl+1 for Daily Brief, Cmd/Ctrl+2 for Recurring-failure, etc. Configurable per-user. |
| Skill version | Small text on hover — e.g. "Daily Brief v1.0" — links to skill changelog (Phase 1.5; MVP shows version only) |
| Disabled state | When BYOK provider unreachable (FR20), all skill buttons disabled with tooltip: "Kondukt unavailable — your LLM provider returned {status}." Live Board + Kanban remain functional. |

| Skill | Default keyboard shortcut | Brief description |
|---|---|---|
| **Daily Brief** | Cmd/Ctrl+1 | Morning situation report — fleet overnight state, changes since last shift, recommended focus |
| **Recurring-failure pattern report** | Cmd/Ctrl+2 | Cross-fleet pattern analysis on a specific subsystem or train cohort |
| **Vendor meeting prep doc** | Cmd/Ctrl+3 | Vendor-facing report (Stadler / Siemens / Knorr-Bremse escalation) with allowlisted internet research |

| State | Trigger | Visual |
|---|---|---|
| Idle (empty canvas) | Roland just navigated to Kondukt; no prior session content | Empty canvas with placeholder text; skill buttons enabled; status: HEALTHY |
| Skill running | Skill invoked, running | Invoked skill button highlights with spinner; canvas shows skeleton placeholder; status text near canvas shows elapsed time |

| First-time user (no BYOK key) | Operator hasn't completed BYOK setup (scenario 15) | Skill buttons disabled with tooltip; banner: "Configure your BYOK key first. [Go to setup]" → links to scenario 15 |

## Behaviour Notes

- **Skill invocation is an operator action** logged per FR32 — `action_type='kondukt_skill_invoked'`, `skill_name='daily_brief'`, IdP, timestamp.
- **Recommendation triplet starts at invocation.** The Envelope is partially populated at invocation (skill_invoked, operator_id, timestamp, harness_version, model_provider, model_version). Provenance Bundle + Validation Receipt are appended when the LLM call + validator complete (next sub-page 13.2 or 13.4).
- **Slash-command parsing** is forgiving — `/daily-brief`, `/db`, `/daily brief` (with space) all map to the same skill. Phase 4 confirms alias list.
- **Keyboard shortcuts are per-user configurable** — defaults Cmd/Ctrl+1–4 mapped to the four MVP skills. Roland can remap from BYOK settings (scenario 15).
- **Performance:** view render <500ms; skill button click → spinner visible <100ms; total skill run typical <20s (LLM-bound, depends on customer's BYOK provider).
- **Multiple skills concurrently** — MVP assumption: one skill at a time per session. Second invocation while one is running cancels the previous. Phase 1.5 may allow parallel.
- **Provider info display** consolidated in token usage strip; full detail in identity menu (Live Board 11.1) and BYOK settings (15).

## Open Questions

- **Layout: split-pane vs vertical stack** — canvas + chat input arrangement. Working assumption: vertical (chat input below canvas). Phase 4 confirms with usability testing.
- **MCP tool call breadcrumb visibility** — surface during skill running, or only in skill inspection panel? Working assumption: visible during running (small text near status indicator) for transparency; Phase 4 confirms exact treatment.
- **Skill button visual density** — 4 buttons fit comfortably at ≥1280; 5+ may overflow. Working assumption: horizontal scroll on overflow; small "all skills" affordance to view registry as a list. Phase 4 confirms.
- **Skill version + changelog visibility** — versions visible on hover working assumption; full changelog as Phase 1.5. Phase 4 confirms whether changelog is MVP-needed.
- **Cancel affordance for in-flight skill** — MVP working assumption: not cancellable (skill runs short). Phase 1.5 if Roland reports need (e.g. accidentally invoked wrong skill).
- **Daily Brief re-invocation behaviour** — if Roland clicks Daily Brief twice in a session, does the second invocation replace the first brief in canvas, append a new brief, or render side-by-side? Working assumption: new brief replaces canvas content (canvas is single-artifact in MVP); previous brief retrievable via "session history" — Phase 1.5. Phase 4 confirms.
- **Chat input vs skill registry visual weight** — both are interaction surfaces. Skill registry is the primary path (codified workflows); chat is the catch-all. Working assumption: skill registry visually more prominent on first view; chat input persistent below. Phase 4 confirms.
