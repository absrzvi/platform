## Overview

**Page Purpose:** Elena drilled into the door motor card. The expanded view shows full technician-depth detail (fault code, port + MAC, last-known-good state, replacement part fitted, test result) and surfaces the ACK action with pre-populated metadata. Tapping ACK fires IdP auth (04.5) before the write commits.

**User Situation:** Elena just fitted part 78-2299-A and ran the test (PASS). She wants to log her ACK — one tap to log it, IdP auth per-action, done.

**Success Criteria:**
- Full technician detail visible — code, port, MAC, LKG, replacement part, test result
- ACK action surface shows pre-populated detail (from IntentPacket); Elena can review without typing
- ACK button is prominent; "Add note (optional)" inline if Elena wants to attach context
- Tap ACK → IdP auth challenge (04.5); no premature commit

**Entry Points:**
- Tap action button "ACK" on the card stack (04.3)
- Tap card body → read-only preview (same content; ACK button visible at bottom of preview)

**Exit Points:**
- Tap "ACK — log resolution" → [04.5-idp-auth-per-action](../04.5-idp-auth-per-action/04.5-idp-auth-per-action.md)
- Collapse → returns to card stack
- (No DEFER option here — this is the CRITICAL ACK surface; for WARNING DEFER, see [04.7](../04.7-card-detail-with-defer-reason/04.7-card-detail-with-defer-reason.md))

## Technical Notes

- **ACK payload assembly:** the client prepares `{fault_id, vehicle_id, role_context, action_type: "ACK", canonical_ack_text, optional_note}` and passes to 04.5; the IdP auth there enriches the payload with `{idp_token, idp_subject, idp_issuer}` before writing.
- **Pre-populated values:** server-side IntentPacket-derived fields are read-only (test result, LKG, fault code). Operator-entered fields (part number, optional note) are editable.
- **Part number entry pattern:** Phase 4 spec — current MVP assumes part number is in IntentPacket (telemetry from depot diagnostic kit). Vendor-portal paste fallback is open question.
- **Canonical ACK text generation:** server template per fault category + resolution fields. Future: LLM-generated text for novel cases (post-MVP).
- **No identity yet:** Elena's role context is in client state; identity fires only when she taps ACK and goes through 04.5.

## Open Questions

| # | Question | Context | Status |
|---|----------|---------|--------|
| 1 | Part number entry — pre-populated from IntentPacket or manually entered? | Most depots have diagnostic kits that emit part-fitted events; for those, pre-populated. For vendors that don't, manual entry with vendor-portal-paste pattern. Recommend: pre-populate where possible, manual fallback. | 🔴 Open |
| 2 | Optional note discarding on collapse — does Elena lose work? | Persona F3 — minimal overhead. Recommend: if note has >0 chars, prompt "Discard note?" before collapse. | 🔴 Open |
| 3 | Canonical ACK text — should Elena see the *raw template* or only the rendered text? | Rendered text only; raw template is internal. The optional note is her way to deviate from canonical. | 🟢 Resolved: rendered text only |
| 4 | Long-press to copy detail values — is this necessary? | Persona W4 — technician convenience. Cheap to add. Recommend: yes, on all mono-rendered values (codes, MACs, timestamps). | 🟢 Resolved: long-press copy on mono values |
| 5 | Multiple Electrical technicians on same vehicle — race condition on ACK? | Server idempotency on `(fault_id, action_type)` — second ACK on the same fault gets a 409 with the first one's details; Elena sees a friendly message "Already acked by {other_tech} at {time}" and the card refreshes to already-acked state. | 🟢 Resolved: idempotent + race-condition UX |
