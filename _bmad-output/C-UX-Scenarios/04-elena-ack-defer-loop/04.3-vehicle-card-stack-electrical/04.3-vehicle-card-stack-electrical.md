## Overview

**Page Purpose:** Elena tapped Unit 4815. She sees the vehicle's open electrical-domain faults — sorted by severity, role-filtered to Electrical, **with brake-wear alerts absent** (not hidden, *not present* per persona W1 + UX decisions Q1). Each card has its pending action (ACK or DEFER) inferred from severity + state.

**User Situation:** Elena has Unit 4815 selected. She sees a door motor card (pending ACK — she just fitted the part) and a camera firmware card (pending DEFER — non-blocking, schedule with next maintenance window).

**Success Criteria:**
- Cards filtered to Electrical role only — brake wear, mechanical-only faults absent
- Each card prompts the appropriate action via its primary CTA
- Card depth = technician (fault code, port, last-known-good, diagnostic step) — different lens from Klaus's 03.2
- Cross-domain faults (e.g., camera-power) appear here if tagged Electrical too (multi-tag `role_relevance`)

**Action button rules:**

- **CRITICAL severity**: only ACK shown (DEFER blocked — CRITICAL faults must be resolved, not deferred)
- **WARNING severity**: ACK + DEFER both shown
- **INFO severity**: ACK + DEFER both shown; DEFER is more common
- **Already-acked**: action buttons hidden; tap-into-card shows the resolution chain (read-only)
- **Already-deferred**: action buttons hidden; tap shows the deferral detail (read-only)

## Technical Notes

- **Server-side role filtering:** `GET /api/vehicle/{vehicle_id}/cards?role=electrical` returns cards where `role_relevance[]` includes "electrical". `role_relevance` is multi-tag (per pre-pilot blocker); cross-domain faults appear in multiple roles' lists.
- **Card depth applied server-side:** technician-lens content (fault code, port, MAC, LKG, diagnostic) generated at brief-generation time per role. Klaus and Elena receive different `summary` text for the same fault from the same IntentPacket.
- **Action button availability:** server-asserted based on severity + state — client doesn't compute this independently. The server knows whether a card is CRITICAL or WARNING; it returns the allowed actions per card.
- **DEFER blocked on CRITICAL:** schema-level constraint; the server simply doesn't return DEFER as an allowed action for CRITICAL cards. If a technician needs to escalate something CRITICAL beyond their authority, they ACK with a free-text note (out of MVP, see 04.4 open questions).
- **Brake-wear absence:** pre-pilot taxonomy decision — `role_relevance[]` for brake-wear faults is set to mechanical-only at the ingest layer. Persona W1 + UX decisions Q1 explicitly state brake alerts are *not present* on Electrical view, not just hidden.
