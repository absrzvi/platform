## Overview

**Page Purpose:** Verify Elena's identity *per-action* on a shared device. Role was shared (tablet at Bay 5); identity must not be. Elena taps ACK or DEFER; this modal fires; she authenticates (biometric if available, PIN fallback); the action commits with her IdP attached to the audit row. The next technician at the tablet cannot inherit her identity — every action requires fresh auth.

**User Situation:** Elena tapped "ACK — log resolution" in 04.4 (or "DEFER" in 04.7). The IdP auth modal fires immediately. She authenticates; the action commits; she sees the confirmation (04.6 or 04.8).

**Success Criteria:**
- Modal fires within <300ms of action tap
- Biometric path (Face ID / Touch ID / Android equivalent) completes in <2s when available
- PIN fallback for biometric-unavailable scenarios (~6 digits, on-screen keypad)
- Identity NEVER cached beyond the current action — every ACK/DEFER on shared device re-prompts
- Cancel returns to the originating screen (04.4 or 04.7) with the action UNCOMMITTED
- The action payload prepared in the originating screen is preserved across the auth modal

**Entry Points:**
- Tap "ACK — log resolution" in [04.4-card-detail-with-ack](../04.4-card-detail-with-ack/04.4-card-detail-with-ack.md)
- Tap "DEFER" in [04.7-card-detail-with-defer-reason](../04.7-card-detail-with-defer-reason/04.7-card-detail-with-defer-reason.md)

**Exit Points:**
- Successful auth → action commits, navigates to [04.6-ack-confirmation](../04.6-ack-confirmation/04.6-ack-confirmation.md) or [04.8-defer-confirmation](../04.8-defer-confirmation/04.8-defer-confirmation.md)
- Cancel → returns to 04.4 / 04.7 with prepared payload preserved (Elena can edit and try again)
- Auth failure (biometric / PIN) → in-modal retry up to 3 attempts; then lockout for 30 seconds

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Biometric available + registered | WebAuthn supported + user has registered | Biometric prompt fires immediately on modal open | Authenticate, switch to PIN, cancel |
| Biometric unavailable | WebAuthn not supported or user not registered | Modal opens directly in PIN mode; biometric section absent | Enter PIN, cancel |
| Biometric failed → PIN fallback | WebAuthn assertion failed | "Biometric failed — use PIN instead"; PIN keypad becomes active | Enter PIN, cancel |
| Auth success | WebAuthn or PIN succeeded | Brief success indicator; modal closes; action commits; nav to 04.6/04.8 | None — auto-progress |
| PIN failure (1-2 of 3) | Wrong PIN | Failure message + retry; counter decrements | Retry, cancel |
| PIN failure (3 of 3) → lockout | Three wrong PINs | 30s lockout with countdown; keypad disabled | Wait, cancel |
| Network offline at commit | Auth succeeded but action write failed | Action queues locally with idempotency key; nav to 04.6/04.8 with queued-state UX (mirroring 03.7 offline handling) | None |
| User registered biometric not present | User who registered fingerprint on a different device | Falls back to PIN; offer "Register biometric on this tablet?" post-action (out of MVP scope) | PIN |

## Technical Notes

- **WebAuthn flow:** standard FIDO2 with the ÖBB SSO IdP as the relying party. Tablet registers user credentials at first enrolment (post-MVP scope for the enrolment UI; backend supports). Per-action assertion returns a signed `idp_token` that the platform uses for the audit row.
- **PIN flow:** user's PIN (6 digits) is registered at IdP setup; verified server-side against the ÖBB SSO IdP's PIN endpoint. NEVER stored or verified client-side. Each PIN attempt round-trips.
- **No identity caching:** the `idp_token` from a successful auth is used ONCE for the immediate action commit, then discarded. No session token, no cookie, no localStorage. Identity is per-action by construction.
- **Lockout:** server-tracked per IdP subject (not per device) — three failed PIN attempts anywhere → 30s lockout for that user. Defensive against shared-device PIN-guessing.
- **Payload preservation across modal:** the originating 04.4 / 04.7 keeps the prepared payload in local component state; the modal lifecycle doesn't touch it.
- **Persona F2 absolute rule:** the audit row's `idp_subject` is ALWAYS the user who completed THIS specific auth challenge. No exceptions, no shortcuts. Renate's criterion depends on this.
