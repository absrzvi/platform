## Overview

**Page Purpose:** Surface the pre-arrival brief to Klaus 14 minutes before HAFAS scheduled arrival, in his peripheral attention (OS notification), without requiring him to open the PWA proactively. This is the moment that turns "the brief exists" into "Klaus knows the brief is ready."

**User Situation:** Klaus is at the depot office, drinking coffee at 05:50. Unit 4722's HAFAS arrival is 06:04. The platform's pre-arrival job computed the brief at 05:50 (arrival − 14 min) and pushes the notification to Klaus's device.

**Success Criteria:**
- Notification fires within 30s of brief computation completion
- Notification copy is glanceable in <2s — vehicle, fault count, time to arrival
- Tap deep-links to the specific brief (03.2) with the correct `vehicle_id`
- No notification if Klaus is already actively in the PWA on that brief (avoid noise)
- Notification de-duplicates if the same brief re-computes within 5 minutes

**Entry Points:**
- Server-side cron / event trigger: HAFAS scheduled arrival − 14 minutes for each vehicle in Klaus's assigned depot

**Exit Points:**
- Klaus taps notification → [03.2-prearrival-brief](../03.2-prearrival-brief/03.2-prearrival-brief.md)
- Klaus dismisses → no further action; brief still available in the PWA

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Default fire | Server triggered notification per cron | Notification arrives on device | Tap → 03.2 / dismiss |
| Re-computation (brief updated within 5 min of previous push) | Same brief re-emitted | NO new notification — the existing one stays current; in-app brief view auto-refreshes | n/a |
| Brief re-computation >5 min after first push | Brief materially changed (new fault, status shift) | New notification with `tag` field that *replaces* prior notification on most platforms; copy adjusts ("Unit 4722 — brief updated"); same deep-link | Tap → 03.2 |
| Klaus already in the brief view | Klaus has the PWA open on the matching brief | NO notification fires (in-app already showing latest) — the server checks an active-session signal via the PWA's heartbeat | n/a |
| Push permission denied | Klaus declined push permission | NO notification; the brief is still computed and available in the PWA — push is the *prompt*, not the *only* delivery path | n/a |
| Offline device | Klaus's phone offline at fire time | Web Push payload queues at the FCM/APNs side; delivered when device reconnects (if within OS TTL — typically 1–7 days). Stale notifications (arrival time already past) are suppressed by a server-side TTL check before queuing | n/a |

## Technical Notes

- **Web Push API:** standard W3C Push API. Service worker registers; subscription stored server-side keyed by Klaus's IdP. Server-side cron fires the notification 14 minutes before HAFAS scheduled arrival.
- **Deep-link handling:** PWA service worker handles `notificationclick` event; opens the deep-link URL in the existing PWA window if present, else opens a new one. Source=push query param is captured by the PWA on entry and emitted to the server as a `pre_arrival_review_opened_via_push` event.
- **Active-session suppression:** PWA emits a heartbeat (every 30s) when on the brief view for a specific vehicle. Server check: if heartbeat fresher than 60s for the same vehicle, suppress the push (in-app already current).
- **Tag-based replacement:** Web Push `tag` field uses `pre-arrival-{vehicle_id}` so updates collapse onto the existing notification. iOS Safari respects this; Android FCM does too.
- **Server-side TTL:** if HAFAS arrival is already past at the moment a notification would deliver (because device was offline), suppress. The brief in-app is still there for retrospective review, but the push is not relevant any more.
- **Permission UX:** the PWA requests push permission contextually (first time Klaus opens the depot briefing PWA after IdP login) with an in-app pre-prompt explaining the value. The OS-level permission prompt only fires after Klaus confirms in the pre-prompt.

## Open Questions

| # | Question | Context | Status |
|---|----------|---------|--------|
| 1 | 14-minute pre-arrival trigger — what if the brief isn't ready 14 min before (IntentPacket arrived late)? | Recommend: fire as soon as brief is ready, with adjusted copy ("HAFAS arrival 06:04 (8 min) · brief ready"). Don't suppress if the trigger missed. | 🔴 Open |
| 2 | Push permission denied — should the PWA periodically re-prompt? | Recommend: no — respect the user choice; the in-app surfaces still work. An explicit "Enable notifications" affordance in Klaus's settings. | 🟢 Resolved: no re-prompt; settings affordance |
| 3 | Multi-vehicle scenarios — Klaus is on shift, three vehicles arrive within 30 min | Each fires its own notification with distinct tag. iOS/Android group them by app. Klaus sees a grouped notification with "3 briefs ready"; tapping expands. | 🟢 Resolved: per-vehicle tags + OS grouping |
| 4 | Sound + haptic — silent push or with sound? | Recommend: respect device-level Do Not Disturb / notification sound settings; default to platform-defined (gentle ping). Persona F2 (no pressure) — but the notification is the gentle nudge that creates the >80% review rate; it must be noticeable. Default sound on, configurable in PWA settings. | 🔴 Open |
| 5 | German translation of notification copy | Pre-pilot blocker — same scope as 03.5/03.6/03.7. | 🔴 Open |
| 6 | What if HAFAS arrival time changes after the push fired? | The brief detail view shows the latest HAFAS time; the notification (already delivered) is stale. Acceptable — the notification is a prompt, the brief is authoritative. | 🟢 Resolved: notification stale is acceptable |
