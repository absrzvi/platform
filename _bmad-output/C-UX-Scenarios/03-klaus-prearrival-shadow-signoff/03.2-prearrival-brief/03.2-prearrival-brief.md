## Overview

**Page Purpose:** Klaus's first view inside the PWA after tapping the push notification. Vehicle-level card stack showing all faults sorted by severity, role-filtered to ECM Manager (executive-summary depth, plain English). Klaus scans the brief in <30 seconds before walking to the bay.

**User Situation:** Klaus opened the PWA from the push notification. He has 14 minutes (or less, if the push fired late) before Unit 4722 arrives. He needs the *story* of the vehicle — what failed, what was resolved, what was deferred — before he walks to Bay 3.

**Success Criteria:**
- Vehicle identity + HAFAS context visible at top
- All faults sorted by severity (CRITICAL → WARNING → INFO → RESOLVED), newest first within tier — same Sprint 1 sort rule
- Card depth = executive summary (plain English, no fault codes); role = ECM Manager applied automatically (stored preference)
- "Record sign-off" affordance visible at top right (always reachable, but not pre-emptive)
- Initial render <1.5s on mobile

**Entry Points:**
- Tap from [03.1-push-notification](../03.1-push-notification/03.1-push-notification.md) (deep-link with `source=push` tracking)
- Direct URL `/brief/{vehicle_id}` (Klaus opens PWA, navigates manually)
- Vehicle list (in-app, not yet specified for MVP — out of Sprint 3 scope)

**Exit Points:**
- Tap a fault card → [03.3-card-detail-critical](../03.3-card-detail-critical/03.3-card-detail-critical.md) (or 03.4 for warning)
- Tap "Record sign-off" → [03.5-return-record-entry](../03.5-return-record-entry/03.5-return-record-entry.md)
- Identity menu → settings, sign out

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Default — brief ready | Brief loaded, vehicle not yet arrived | All sections visible, count-down live | Tap card, record sign-off, navigate |
| Brief refreshing | Server pushed update (new fault, ACK arrived) | Subtle toast top of feed: "Brief updated — {field} changed"; affected card highlights briefly | Tap toast to scroll to card |
| Vehicle docked | HAFAS arrival time passed, vehicle docked | HAFAS line updates ("Unit docked at {time}"); rest of view unchanged | Same |
| Vehicle departed | HAFAS state shows departure | Banner top of feed: "Unit {unit_id} departed at {time}" — brief becomes a retrospective view; sign-off still possible (records as retroactive) | Same |
| No faults | IntentPacket history shows no resolved/unresolved faults | Empty state in feed: "No faults to review. Vehicle ready for clearance." + Record sign-off CTA prominent | Record sign-off |
| Loading | Initial load <1.5s | Skeleton: context placeholder + 3 card placeholders | None |
| Stale brief (>15 min since generation, no updates) | Klaus came back to a brief that's been sitting | Subtle indicator on Brief generated line: "Brief generated 05:50 UTC · refresh" (tap to refresh) | Refresh, tap card, record |
| Offline | Device offline after opening | Cached brief renders; banner "Offline — last updated {time}"; record sign-off still works (queues locally per 03.7 spec) | Tap card (cached), record (queues) |

## Technical Notes

- **Brief data shape:** `GET /api/brief/{vehicle_id}` returns `{vehicle_id, depot_id, bay_id, hafas_arrival_at, hafas_state, brief_generated_at, role_filtered_faults: [{fault_id, severity, system, location, summary, resolution_status, metadata, role_relevance[]}], summary_counts}`
- **Server-side role filtering:** the `role_filtered_faults` array is already filtered for ECM Manager role at the server before delivery — no client-side filtering of `role_relevance[]`. The role param comes from Klaus's IdP claim + stored preference.
- **Card depth applied server-side:** the executive-summary text + plain-English summary are generated per fault per role at brief-generation time. Klaus's brief and Elena's brief see *different text* for the same fault, sourced from the same IntentPacket.
- **HAFAS count-down:** client-side timer updates every 30s; falls back to "arrival was {time}" once passed.
- **SSE for brief updates:** PWA subscribes to `/api/brief/{vehicle_id}/events` while view is open; updates trigger the toast + affected-card-refresh.
- **Sort order:** server pre-sorts faults per the Sprint 1 LOCKED rule (severity-banded, newest-first within tier). LOW band would not appear here (this is the brief feed, not the triage feed); WARNING is the lowest tier shown in MVP.
- **Performance contract:** initial render <1.5s on a mid-range Android device — measured.
