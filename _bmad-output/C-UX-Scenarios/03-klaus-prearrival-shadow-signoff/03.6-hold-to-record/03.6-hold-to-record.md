## Overview

**Page Purpose:** The defining trust gesture of the platform — 3s hold to initiate, 5s countdown to commit, single atomic write at countdown expiry. Identical specification to Stefan's [01.6-mark-executed](../../01-stefan-crisis-redeployment/01.6-mark-executed/01.6-mark-executed.md) gesture and to the design-system [hold-to-record spec](../../../D-Design-System/00-design-system.md#gesture-hold-to-record-klaus-shadow-sign-off-stefan-mark-executed-analog).

This page documents the *Klaus context-specific* variant — same gesture, mobile-touch primary, German haptic conventions, IdP re-auth path.

**User Situation:** Klaus has read the framing in 03.5 and chosen to record. He presses and holds the button. The 3-second hold initiates the gesture. The 5-second countdown gives him one last chance to cancel by releasing. At expiry, the atomic write fires once.

**Success Criteria:**
- 3s hold + 5s countdown timing identical to Stefan's 01.6 (locked in design system)
- Release at any point during 3s hold or 5s countdown cancels with no write
- At countdown expiry, exactly one atomic write fires; idempotency key generated at hold initiation
- Cancellation announces clearly (no ambiguity about whether write fired)
- Motion-reduced accessibility variant available (countdown without radial animation)

**Entry Points:**
- onTouchStart on the hold button in [03.5-return-record-entry](../03.5-return-record-entry/03.5-return-record-entry.md)

**Exit Points:**
- Atomic write fires at countdown expiry → [03.7-signoff-success](../03.7-signoff-success/03.7-signoff-success.md)
- Release during 3s hold → cancels, returns to 03.5 idle
- Release during 5s countdown → cancels, returns to 03.5 idle (with explicit "Cancelled — no record written" toast)
- Network/identity failure mid-gesture → error state, no write, returns to 03.5 with explanation

#### State: Idle

**OBJECT ID:** `hold-record-button-idle`

| Property | Value |
|----------|-------|
| Component | Filled primary button |
| Label | "Hold to record (3s)" |
| Behavior | onTouchStart → transitions to Initiating |

#### State: Initiating (during 3s hold)

**OBJECT ID:** `hold-record-button-initiating`

| Property | Value |
|----------|-------|
| Component | Filled primary button with progress bar (left-to-right) overlaid |
| Label | "Initiating…" |
| Behavior | onTouchEnd before 3s → cancels, returns to Idle; haptic vibrate (subtle) at the 3s mark when transitioning to Counting Down |
| Visual progression | Bar fills smoothly over 3000ms; if user holds beyond 3s, transitions to Counting Down state |
| Motion-reduced variant | Replace the smooth bar with discrete tick marks (3 ticks) appearing at 1s intervals; the button colour stays solid |

#### State: Counting Down (5s)

**OBJECT ID:** `hold-record-button-counting-down`

| Property | Value |
|----------|-------|
| Component | Button transforms to a circular gauge — radial countdown ring + large central number |
| Label | "5" → "4" → "3" → "2" → "1" (large central digit) + caption "Hold to record · cancel by releasing" |
| Behavior | onTouchEnd anywhere during countdown → CANCELS, returns to Idle with explicit "Cancelled — no record written" toast |
| Visual progression | Radial ring depletes (clockwise, ~360deg over 5000ms); central digit updates each second; modal backdrop dims to focus attention |
| `aria-live` | The countdown number announces each second ("5", "4", "3"…) for screen readers |
| Motion-reduced variant | No radial ring; large central digit changes each second with a subtle background tint pulse; same 5s duration |

#### State: Writing (at countdown expiry)

**OBJECT ID:** `hold-record-button-writing`

| Property | Value |
|----------|-------|
| Component | Button transforms briefly (≤1s) to a "Writing…" state |
| Label | "Recording…" |
| Behavior | Atomic write fires; idempotency key generated at hold initiation is sent with the write; on response, transitions to Success state |
| Visual | Brief flash, then transition to Success state |

#### State: Success (post-write)

**OBJECT ID:** `hold-record-button-success`

| Property | Value |
|----------|-------|
| Component | Button briefly shows success checkmark + green flash |
| Label | "Recorded ✓" |
| Behavior | After ≤1s, navigates to [03.7-signoff-success](../03.7-signoff-success/03.7-signoff-success.md) |

## Page States

(Page states here = gesture states; covered in Sections above. Below: failure / edge cases.)

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Cancel during hold | onTouchEnd <3000ms after touchStart | Button returns to Idle; no toast (the gesture didn't pass the initiation threshold) | Re-hold or cancel |
| Cancel during countdown | onTouchEnd between 3000ms–8000ms after touchStart | Button returns to Idle; explicit toast "Cancelled — no record written" (live region) | Re-hold or cancel |
| Write success | Atomic write returned 2xx | Brief success flash; nav to 03.7 | None — auto-progress |
| Idempotency hit (409) | Server returned 409 with same idempotency key — duplicate write attempt | Treated as success (the original write succeeded); nav to 03.7 | None |
| Write error (5xx, timeout) | Server returned error | Button transforms to error state with red flash + "Recording failed — retry?" inline; underlying modal stays open; user can re-attempt or cancel | Retry hold, cancel |
| Identity expired mid-gesture | IdP token expired between touchStart and atomic write | Button transforms to "Re-authenticate to continue" state with biometric prompt; on success, gesture resumes from where it was interrupted | Re-auth, cancel |
| Network offline at write | Device offline when atomic write would fire | Write queued locally with idempotency key; nav to 03.7 with "Queued, will sync on reconnect" state | None — auto-progress |
| Keyboard-equivalent | User holds Enter for 3s, then countdown auto-progresses | Same visual phases as touch; release Enter at any point cancels | Hold Enter, release to cancel |

## Technical Notes

- **Idempotency key generation:** at onTouchStart (hold initiation), client generates `crypto.randomUUID()` and stores in component state. Same key is sent if the user retries after a 5xx error — guaranteeing exactly-once semantics on the server side.
- **Atomic write request:** `POST /api/audit/shadow-events` with payload `{vehicle_id, paper_sign_off_at, recorded_at, hold_started_at, countdown_completed_at, atomic_write_requested_at, idempotency_key, idp_token}`. Server enforces append-only on `ecm_signoff_events`.
- **Hold timing precision:** measured via `performance.now()` (sub-ms precision); the 3000ms / 5000ms boundaries are absolute, not approximated. UI animations run on `requestAnimationFrame`.
- **Cancel-on-release semantics:** release MUST cancel during both the 3s hold and the 5s countdown. The only way to commit is to hold continuously until countdown expiry. This is the core trust gesture — Klaus must explicitly commit, never accidentally trigger.
- **Motion-reduced fallback:** `prefers-reduced-motion: reduce` swaps radial ring → discrete countdown digit + subtle pulse. The total 3s + 5s = 8s timing is preserved.
- **Keyboard equivalent:** long-press Enter (sustained keydown). The same 3s + 5s + atomic-write semantics apply. This makes the gesture WCAG-compliant for non-touch users.
- **Backdrop dim during countdown:** the modal's parent backdrop dims slightly during the 5s countdown to focus Klaus's attention on the gesture. This is *not* a forced focus — Klaus can still tap Cancel or release.
- **Persona F2 contract — no pressure cues:** no audio chime, no haptic during the count-down progression. The only haptics are subtle: at initiation (touch registered) and at success (write committed). Persona F2 explicitly: no urgency theatre.

## Open Questions

| # | Question | Context | Status |
|---|----------|---------|--------|
| 1 | If Klaus accidentally releases during the 5s countdown, should there be a "Restart hold" affordance, or back to a full re-tap? | Current: full re-tap (returns to Idle). Alternative: a "Restart hold" button appears for 3s after cancel. Recommend: full re-tap — the explicit re-initiation is the trust-building muscle memory we want. | 🟢 Resolved: full re-tap |
| 2 | Should the central countdown number stay visible after the gesture completes, briefly? | Recommend: replace with checkmark immediately at countdown expiry; the success state is the natural landing place. | 🟢 Resolved: checkmark on expiry |
| 3 | What if Klaus tries to navigate away mid-countdown (e.g., taps a notification)? | The modal is focus-trapped; navigation events are blocked during countdown. If the OS forces a navigation (incoming call), the gesture cancels with no write; on return, modal is in Idle state. | 🟢 Resolved: focus-trap; OS interrupts cancel cleanly |
| 4 | German haptic conventions | iOS / Android haptic APIs are language-neutral. No locale-specific behaviour needed. | 🟢 Resolved: platform-native haptics |
| 5 | Subtle haptic at the 3s → countdown transition — does this read as "pressure" (violating F2)? | The haptic is a confirmation that the gesture has moved phases, not urgency. Recommend: keep it (single tick, not pattern); test in usability sessions. | 🔴 Open |
| 6 | What if the server takes >2s to respond to the atomic write? | The Writing state should not last >1s visually; if the network is slow, the state extends with a subtle "Still recording…" message after 2s. Recommend: backdrop indicates progress without alarming. | 🔴 Open |
