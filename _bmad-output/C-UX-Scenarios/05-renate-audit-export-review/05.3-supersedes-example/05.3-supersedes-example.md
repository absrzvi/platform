## Overview

**Page Purpose:** Demonstrate the platform's correction model on a worked example: how do you fix an audit row that contained an error, without mutating the past? Renate sees both rows visible at once, the linkage between them, the hash-chain proof that the original was not modified, and a "Reason for correction" field on the correcting row.

This page exists specifically because Renate's "F1 — compliance theatre" fear demands a *concrete demonstration* of the supersedes model, not a description of it.

**User Situation:** Martin walked Renate from 05.2 to a prepared example. The example: Klaus accidentally signed off Unit 4831 with an incorrect part number; he corrected it 4 hours later. Both rows visible.

**Success Criteria:**
- Both rows visible simultaneously (no need to scroll between them)
- Linkage between rows visually obvious — the correcting row points to the original
- The original row's hash is preserved and visible (proof of non-mutation)
- The correcting row's hash includes the original's hash in its chain (proof of chain continuity)
- "Reason for correction" is a prominent field, not buried

**Entry Points:**
- "Show correction example" affordance from 05.2 footer
- Direct URL `/audit/supersedes-example/{original_row_id}` (Martin's prepared URL for the manday-15 review)
- Direct URL `/audit/supersedes-chain/{any_row_id}` (general — shows the supersedes chain for any row that's been corrected)

**Exit Points:**
- "Open this correction in detail" → [05.4-supersedes-detail](../05.4-supersedes-detail/05.4-supersedes-detail.md)
- "Back to audit view" → returns to [05.1-audit-export-view](../05.1-audit-export-view/05.1-audit-export-view.md) for the most recent row in the chain

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Default | Chain has exactly 2 rows (1 original + 1 correcting) | Two stacked panels, arrow between, hash chain proof block | Verify chain, open details |
| Multi-step chain | Chain has 3+ rows (correction-of-correction-of-original) | Stacks all rows in chronological order, each with linkage arrow to next; supersedes_id link from latest points all the way back through the chain | Same actions + chain depth indicator |
| Verifying chain integrity | After "Verify chain integrity" clicked | Spinner inline; on completion, green check + "Chain verified at {timestamp}" or red ! + reason for failure | Re-verify, escalate (if failed) |
| Chain integrity failure | Recomputed hash doesn't match | Red banner across whole page: "Chain integrity failure — escalate to ops"; both row panels get red border treatment | Re-verify, escalate, navigate |
| Loading | Initial load | Skeleton: two stacked panel placeholders + arrow placeholder | None |

## Technical Notes

- **Chain fetch:** `GET /api/audit/shadow-events/{row_id}/chain` returns the full supersedes chain for the row (all ancestors via `supersedes_id` and all descendants via reverse-lookup), in chronological order, each row including its HMAC + prev_row_hash + this_row_hash.
- **Chain integrity recomputation:** server-side walk; the proof statement is server-asserted, the client renders.
- **Multi-step chains:** rendered as N panels stacked; each panel labelled "ROW {seq} (Original)" / "ROW {seq} (Correcting Row {prev_seq})". The latest row carries the ACTIVE badge; all others carry SUPERSEDED.
- **Print stylesheet behaviour:** linkage arrows become "↓ (supersedes Row N)" textual annotations; chain proof block renders as a separate page. Audit committee can read the chain end-to-end on paper.
- **Schema constraint on correction_reason field:** ≥ 16 chars enforced server-side; ≤ 1000 chars; UTF-8 only; no HTML rendering on read (display as plain text with newline preservation).
