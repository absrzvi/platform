## Overview

**Page Purpose:** A document-style render of a single ECM shadow audit row for one vehicle on one sign-off event. Renate (or her audit-committee analyst) opens this URL and must be able to read it as a regulatory document, not a UI. Every field that appears on the paper sign-off must be reconcilable on this page; the engineering rigour (HMAC, hash chain, IdP identity) must be visible and verifiable.

**User Situation:** Renate has Martin Lerch's chat message open with the URL for Unit 4722's audit export. She clicks it. She has the paper scan in another tab. She is reading both side-by-side.

**Success Criteria:**
- Document loads in under 2s
- Every paper-sign-off field has a labelled equivalent on this page
- HMAC + hash-chain proof verifiable in-place (or one click to verify)
- Structured JSON export available without leaving the page
- Print stylesheet renders the document as a one-page PDF suitable for audit-committee submission

**Entry Points:**
- Direct URL `/audit/shadow-events/{audit_row_id}` — IdP-authenticated, audit-reviewer role required
- "Open in audit view" link from reconciliation queue [05.5](../05.5-reconciliation-queue/05.5-reconciliation-queue.md)

**Exit Points:**
- Renate moves to side-by-side paper comparison → mental flow continues to [05.2-compare-with-paper](../05.2-compare-with-paper/05.2-compare-with-paper.md) (mostly external — she opens paper scan beside this view)
- Martin shows her supersedes example → [05.3-supersedes-example](../05.3-supersedes-example/05.3-supersedes-example.md)
- Renate navigates to reconciliation queue → [05.5-reconciliation-queue](../05.5-reconciliation-queue/05.5-reconciliation-queue.md)

## Technical Notes

- **Data shape:** `GET /api/audit/shadow-events/{audit_row_id}` returns the full structured row including key timestamps, IdP identity proof, integrity fields (HMAC + prev_row_hash + this_row_hash), source_tools list, fault resolution chain (joined from related tables), and the `record_type` field (`shadow` in MVP, `authoritative` in Phase 2).
- **Integrity verification:** `POST /api/audit/shadow-events/{audit_row_id}/verify` recomputes HMAC and walks the hash chain from this row backward to the most recent verified anchor (typically the previous pilot-day rollup). Server returns `{verified: true|false, anchor_row: ..., walk_length: N}` or `{verified: false, mismatch_at: {row_id, expected_hash, actual_hash}}`.
- **Print stylesheet:** required for MVP. Renate's audit committee will be handed PDFs, not URLs. The stylesheet hides chrome, expands all values to full (no truncation), and uses proper page-break rules.
- **JSON export schema:** stable, versioned (`schema_version: "ecm-audit-v1.0.0"`). The export schema is the contract with downstream documentation systems (BOOM integration deferred but schema is the future-target). Breaking changes require a new schema version, never a silent shape change.
- **HMAC + hash chain verification:** server-side computation; client never holds the HMAC key. The "Verify integrity now" button is essentially "ask the server to re-prove the row" — a one-liner from the client's perspective.
- **IdP issuer rendering:** the `sso.oebb.at` (or equivalent operator-specific) domain is rendered verbatim. Any non-IdP-bound identity is rejected at the schema layer and never reaches this page.

## Open Questions

| # | Question | Context | Status |
|---|----------|---------|--------|
| 1 | Sign-off date row — display as ISO date (`2026-10-12`) or full datetime in operator's local TZ? | Paper sign-off carries just a date; full datetime is engineering-true but visually heavier. Recommend: ISO date primary, with the full UTC datetime of the shadow row available in the Key Timestamps section. | 🔴 Open |
| 2 | "Verify integrity now" — should it auto-verify on page load, or wait for explicit click? | Auto = always shows verified state, reassuring. Click = explicit gesture, more rigorous. Recommend: explicit click for MVP — Renate's audit committee notes the *act of verification*. | 🔴 Open |
| 3 | HMAC + hash truncation length on screen | 16 chars + ellipsis is current spec. Tradeoff: shorter = compact, longer = more glanceable. Print stylesheet shows full. Recommend: 16-char truncation on screen, full on print. | 🟢 Resolved: 16 chars screen, full in print |
| 4 | "Source tools" — should low-confidence source tools be flagged here for Renate? | The RAO that produced this audit row had a `confidence` value. If LOW, Renate should know — modification was forced, the row already encodes that. But this is a *sign-off* row, not an *RAO acceptance* row — different rendering pattern. Recommend: include the original RAO's confidence pill if applicable, decorate accordingly. | 🔴 Open |
| 5 | Integrity verification failure — what is the escalation path? | Failure is a regulatory-grade incident. Recommend: failure rendering AUTO-creates an incident in the platform's ops queue (separate from operator surfaces), notifies Martin Lerch + Renate's IdP within 1h. | 🔴 Open |
| 6 | Multiple shadow rows for the same vehicle/sign-off (supersedes chain) | Renate may arrive at a row that has been superseded by a correction. Should this page redirect her to the most recent row, or render this row with a "superseded by..." banner? Recommend: render this row + banner — the request URL is the authoritative document the audit committee saw. | 🟢 Resolved: render requested row + supersedes banner |
