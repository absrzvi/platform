## Overview

**Page Purpose:** Field-by-field diff comparison between an original audit row and its correcting row, with differences highlighted. Where 05.3 shows the *chain* (both rows + linkage proof), 05.4 zooms into the *delta* (what changed, field by field). Renate uses this to verify the change is precisely what the `Reason for correction` text says.

**User Situation:** Renate clicked the `supersedes_id` link on the correcting row in 05.3 (or "Open Row 2 in detail" from the footer). She wants to see exactly which fields differ and confirm the diff matches the reason.

**Success Criteria:**
- Original (left) and Correcting (right) panes render the same fields in the same order
- Differing fields are highlighted with consistent visual treatment
- Identical fields are visually de-emphasised so the differences pop
- Reason for correction is prominent on the right pane
- A summary line states the field-level delta in plain English ("1 field differs: part_fitted")

**Entry Points:**
- "Open Row 2 in detail" from 05.3
- Direct URL `/audit/supersedes-detail/{correcting_row_id}`

**Exit Points:**
- "Back to chain view" → [05.3-supersedes-example](../05.3-supersedes-example/05.3-supersedes-example.md)
- "Open in audit view" (for either row) → [05.1-audit-export-view](../05.1-audit-export-view/05.1-audit-export-view.md)
- "Open reconciliation queue" → [05.5-reconciliation-queue](../05.5-reconciliation-queue/05.5-reconciliation-queue.md)

## Layout Structure

Two-pane field-level diff. Original on the left, Correcting on the right. Identical field rows are dimmed; differing field rows are highlighted (yellow tint, bold value). A summary line at the top states the delta count. Reason for correction shown as a full-width block below the diff.

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Default | Diff loaded | Two panes, legend, reason block, verification block, footer | Verify, navigate, download |
| Many differing fields | More than 1 field differs | All differing fields highlighted; summary line lists them; if >3 fields, summary truncates with "+N more" and a "Show all" expand affordance | Same actions |
| All-identical (impossible by definition) | n/a — supersedes rows must differ in at least one operational field; if not, the row is rejected at schema level | n/a | n/a |
| Verifying both rows | After "Verify both rows now" clicked | Spinner inline; on completion, green check + "Both rows verified" or red ! + reason | Re-verify, escalate |
| Verification failure | Recomputation mismatch | Red banner across page: "Integrity failure — escalate"; both panes get red border | Re-verify, escalate, navigate |
| Loading | Initial load | Skeleton: two pane placeholders + reason placeholder | None |

## Technical Notes

- **Diff computation:** server-side. `GET /api/audit/shadow-events/{correcting_row_id}/diff` returns `{original_row, correcting_row, differing_fields: [field_names], identical_fields: [field_names], canonical_field_order: [...]}`. Client renders directly from this payload.
- **Canonical field order:** server-asserted to ensure both panes render fields in the same order regardless of underlying storage. Order matches 05.1's section order: title block fields, key timestamps, fault resolution chain, identity, integrity proof, source tools metadata.
- **Identical-field rendering:** dimmed (`text-secondary`), no background; `aria-hidden="false"` (still accessible) but visually de-emphasised so the eye lands on the ⚠ marks.
- **Differing-field rendering:** `text-primary` + semibold + `bg-amber-100`-ish background tint (matching the modification-row warning treatment in 02.5). Mono font for the value.
- **Reason field rendering:** verbatim from `correction_reason`. Newlines preserved (`white-space: pre-wrap`). No HTML rendering. UTF-8 only.
- **Download diff as PDF:** server-side render via the same print-stylesheet pipeline as 05.1's "Download as PDF" — composed for audit-committee use.

## Open Questions

| # | Question | Context | Status |
|---|----------|---------|--------|
| 1 | Which fields are eligible for diff display vs which are hidden? | Some fields are infrastructure (HMAC, hash) — they always differ between rows by design. Show them in the diff or hide? Recommend: hide HMAC/hash from the diff (they're in the verification block), show only operational fields. | 🟢 Resolved: hide HMAC/hash from diff; verification block carries them |
| 2 | What about timestamp fields — they always differ (later signed time on the correcting row)? | Show in diff as ⚠ differing, or treat as expected metadata? Recommend: show in diff (they're operationally meaningful — Renate's audit committee wants to know when the correction happened), don't suppress. | 🟢 Resolved: show timestamps in diff |
| 3 | Multi-step chain diff (correcting-the-correcting) — which two rows are diffed here? | This page diffs a row against its *immediate* predecessor (the row it supersedes). For full chain diff, see 05.3. Recommend: keep 05.4 scoped to immediate predecessor; add a "Diff against original" affordance for chain-of-3+ scenarios. | 🔴 Open |
| 4 | More-than-3 differing fields — when does "+N more" expand? | Default: summary line shows first 3 differing field names, then "+N more". Click to expand the summary line OR scroll to see all (they're all rendered in the panes anyway). Recommend: expand the summary line. | 🔴 Open |
| 5 | "Verify both rows now" — is this a different verification than 05.3's "Verify chain integrity"? | Yes: 05.3 verifies the chain *link* (Row 2's chain hash includes Row 1's hash). 05.4 verifies each row's *own* HMAC independently plus the chain link between them. Both should be available; some redundancy is fine for audit reassurance. | 🟢 Resolved: yes, both verifications available |
