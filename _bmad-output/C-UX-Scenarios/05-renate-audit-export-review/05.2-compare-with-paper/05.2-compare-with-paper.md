## Overview

**Page Purpose:** Help Renate (and her audit-committee analyst) do field-for-field reconciliation between the platform's shadow row and the paper sign-off scan. Currently this is two-tab-juggling; this surface gives her a structured side-by-side mode if the paper scan has been attached (or upload-on-demand if not).

**User Situation:** Renate has 05.1 open. She wants to align it against the paper. In MVP the typical path is "open paper scan in another tab" — but if the paper has been uploaded to the platform alongside the shadow row (depot scan workflow), she can use side-by-side mode here.

**Success Criteria:**
- Side-by-side mode reads naturally — the platform's row on the left, the paper scan on the right
- Field anchors (vehicle, date, signer, fault chain) align visually across both sides
- If no paper scan is attached, the page degrades gracefully to "drag-drop or open in tab" with clear messaging

**Entry Points:**
- From [05.1-audit-export-view](../05.1-audit-export-view/05.1-audit-export-view.md) via "Compare with paper" affordance in the top chrome
- Direct URL `/audit/shadow-events/{audit_row_id}/compare`

**Exit Points:**
- "Open supersedes example" → [05.3-supersedes-example](../05.3-supersedes-example/05.3-supersedes-example.md)
- "Open reconciliation queue" → [05.5-reconciliation-queue](../05.5-reconciliation-queue/05.5-reconciliation-queue.md)
- "Mark reviewed" → records Renate's review event (audit-row-of-the-audit-row) → returns to 05.1

## Technical Notes

- **Paper scan storage:** stored in object storage (S3 or operator-equivalent) with content-addressed keys (`sha256(content)`). The shadow audit row holds a reference to the scan's hash + storage URL. Scan can be re-fetched and re-verified independently of the audit row.
- **Mark-reviewed schema:** `POST /api/audit/reviews` with `{reviewed_audit_row_id, reviewer_idp, reviewed_at, notes?}`. This is an audit row in its own right (audit-row-of-the-audit-row) with the same append-only enforcement.
- **Side-by-side responsive breakpoint:** 1440px is the cut-off. At <1440px, side-by-side becomes pixel-cramped for both the shadow column and the scan viewer. Below 1280px (atypical for audit-reviewer flows), stacked is the only mode.
- **Image viewer dependency:** keep light. `react-zoom-pan-pinch` (~30kb gzip) covers the affordances we need. Alternative: roll a minimal pan/zoom with `framer-motion` if dependency footprint matters.
- **Bundled PDF download:** server-side render — the print stylesheet of 05.1 + the scan image, composed into a 2-page (or N-page if scan is multi-page) PDF. The audit committee receives this, not the URL.

| 1 | Paper scan upload — depot device or audit-reviewer? | If depot device uploads at sign-off time, the IdP-binding is Klaus's (the signer). If Renate uploads later, the IdP-binding is hers (the reviewer). Renate's audit committee may scrutinise this. Recommend: depot device at sign-off, with optional audit-reviewer re-upload if depot missed it (with provenance noted). | 🔴 Open |
| 2 | Image viewer choice — react-zoom-pan-pinch vs roll-your-own | Dependency vs control. Light dep is fine. | 🟢 Resolved: react-zoom-pan-pinch (or equivalent ~30kb library) |
| 3 | Multi-page scan support | Some sign-offs span multiple paper pages. Viewer must support multi-page navigation. | 🔴 Open |
| 4 | Mark reviewed — append-only enforcement | If Renate marks reviewed then later wants to revise her review, follow supersedes_id pattern (matching the platform's core correction model). | 🟢 Resolved: supersedes_id |
| 5 | Print stylesheet for bundled PDF — server-side vs client-side? | Server-side via headless Chrome is more reliable (single-source truth for layout); client-side via window.print + html-to-pdf is lighter. Recommend: server-side because PDFs go to audit committees and consistency matters. | 🔴 Open |
