## Overview

**Page Purpose:** Display the full story of a single paper-vs-shadow divergence: when it was detected, what differed, why, how it was resolved (or why it remains unresolved), who resolved it, and what its impact on the Phase 2 trust composite is. This is the page Renate spends time on; her audit committee will ask her to walk through one of these.

**User Situation:** Renate clicked a divergent row in the reconciliation queue (05.5). She wants the complete chronology and the resolution narrative. If divergence is unresolved, she wants to know what's outstanding and who's working on it.

**Success Criteria:**
- Full chronology of detection → triage → resolution (or unresolved status) visible on one page
- Root cause categorisation visible (reason code) with free-text explanation
- Both paper and shadow row accessible from this page
- Resolver IdP and any related correction (supersedes_id) linked
- Phase 2 readiness impact clearly stated
- Print stylesheet renders this as a single-page audit artefact

**Entry Points:**
- Click on a divergent row in [05.5-reconciliation-queue](../05.5-reconciliation-queue/05.5-reconciliation-queue.md)
- Direct URL `/audit/divergence/{divergence_id}`
- Ops-watch alert link (when a divergence is flagged as above-threshold)

**Exit Points:**
- "Back to queue" → [05.5-reconciliation-queue](../05.5-reconciliation-queue/05.5-reconciliation-queue.md)
- "Open paper sign-off" → opens the paper scan viewer (05.2 surface, single-pane mode)
- "Open shadow row" → [05.1-audit-export-view](../05.1-audit-export-view/05.1-audit-export-view.md)
- "Open correcting row" (if divergence resulted in a correction) → [05.3-supersedes-example](../05.3-supersedes-example/05.3-supersedes-example.md) or [05.1-audit-export-view](../05.1-audit-export-view/05.1-audit-export-view.md)

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Resolved (within threshold) | Status = Resolved AND all threshold checks pass | All ✓ icons green; Phase 2 Impact section all green | Download, navigate |
| Resolved (above threshold) | Status = Resolved BUT TTR exceeded threshold OR pilot-week unresolved-count was exceeded at time of resolution | TTR row shows red ⚠; Phase 2 Impact section flags affected rules with explanation | Download, navigate; flag for committee review |
| Unresolved | Status = Unresolved | Header status badge shows red; Phase 2 Impact section flags unresolved-time clock running; "Resolver" section says "Pending resolution" with "Open ops queue" link | Investigate, escalate, return to queue |
| Escalated | Resolver type = "escalated to ops" | Header shows orange escalation badge; chronology includes the escalation event; "Open ops queue" link prominent | Investigate, follow ops link |
| Auto-resolved | Resolution type = "auto" | Resolver = "Auto-reconciler"; resolution-method explained ("network sync caught up; shadow eventually matched paper") | Read, navigate |
| Loading | Initial load | Skeleton: header, chronology placeholder, sections placeholders | None |

## Technical Notes

- **Data shape:** `GET /api/audit/divergence/{divergence_id}` returns `{divergence_id, vehicle, sign_off_at, status, chronology[], reason_code, explanation, resolver, resolved_at, resolution_type, related_artefacts: {paper_url, shadow_row_id, correcting_row_id?}, phase_2_impact: {ttr_seconds, threshold_ttr_seconds, pilot_week_unresolved_count, threshold_unresolved_count, ...}}`
- **Chronology computation:** server-side join from paper-upload events, shadow-write events, reconciliation-detection events, correction-filing events. Events presented in UTC.
- **Phase 2 impact computation:** server-side. Threshold rules read from pilot config; impact flags computed per divergence.
- **Print stylesheet:** the page is the artefact. Renate downloads this PDF for her audit committee. Must render as single-page (or 2-page max) clean document; chrome hidden; chronology as a numbered list; URLs rendered as footnotes for paper readability.
- **Escalation surface:** out of MVP scope for the full ops workflow, but the link target (`/ops/incidents/{divergence_id}`) is reserved and Renate can follow it to see the ops-side status.
- **No edit affordances on this page in MVP:** the page is read-only for the audit-reviewer role. Resolutions are filed by the operator (Klaus filing a correction) or ops (annotating escalations) elsewhere; this page surfaces the result.
