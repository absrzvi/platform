## Overview

**Page Purpose:** When Roland clicks a component row in the train detail card (12.3 primary disclosure), the row reveals **secondary disclosure** inline below: MTTR (harness-computed from landside-observable Stadler alarm IntentPacket arrival timestamps — `t_close − t_open`) + recurring-failure history (incidents per component over time) + last 24h packet trace (compact log lines) + vendor portal deep-link (pre-filled with the component's MAC).

This is FR10 secondary disclosure — the diagnostic depth Roland needs to **investigate** a component, not just identify it (12.3 primary disclosure handles identification). Secondary disclosure replaces the vendor-portal-lookup + Power-BI-historical reconciliation + Excel-recurring-failure-tracking friction with a single inline reveal.

**User Situation:** Roland is on 12.3 with train 4736-101 expanded. He clicked the CCTV — Überw. Kamera A7 row. Secondary disclosure reveals inline below the row.

**Success Criteria:**
- MTTR visible per component, sourced from harness-computed `t_close − t_open` per FR4 — NOT from Stadler's internal MTTR field
- Recurring-failure history visible as a compact timeline or list (incidents per component over a configurable window, default 90 days)
- Last 24h packet trace visible as compact log lines (≤10 most recent, with "view full trace" affordance)
- Vendor portal deep-link generates the correct URL with this component's MAC pre-filled (per-vendor URL template)
- Reveal is inline (not modal); other component rows remain visible
- Multiple component rows can have secondary disclosure revealed simultaneously
- Operator interaction logged per FR32 — secondary disclosure expansion is a `card_secondary_disclosed` event

**Entry Points:**
- Component row click in 12.3
- Keyboard: Enter on focused component row
- Direct deep-link `?train={id}&detail=expanded&component={component_id}&secondary=expanded` (rare; from support ticket investigation)

**Exit Points:**
- Click component row again → secondary collapses (back to 12.3 primary-only state for that component)
- Click "view full trace" → opens a modal or new tab with the full 24h+ packet trace (modal preferred)
- Click vendor portal deep-link → opens vendor portal in new tab; URL pre-filled with component MAC
- Click Replay archive deep-link (if Kondukt recommendation is linked to this component) → opens Replay archive in new tab (scenario 05.7 surface)
- Click another component row → reveals secondary on that row too (no exclusivity)
- Esc → collapses all secondary-disclosure revealed in the current session for this card

## Components

### MTTR Block

| Element | Content | Source |
|---|---|---|
| Average MTTR | "Average across last N incidents: X h Y m" — N is the count of resolved incidents in the lookback window (default 90 days, configurable per-card) | Computed: average of `(t_close − t_open)` across resolved incidents for this component |
| Formula footnote | Brief inline explanation: "t_close = alarm-cleared IntentPacket arrival; t_open = alarm-raised IntentPacket arrival" — surfaces FR4 sourcing rule | Static |
| Active incident MTTR | Currently-active incident shows "ongoing · X h Y m" — incrementing live | Computed from current time and active `t_open` |
| MTTR trend indicator | Small arrow / sparkline showing whether MTTR is trending up (↑) or down (↓) over the lookback window | Computed from incident series |

**Note explicitly:** the harness does NOT read Stadler's internal MTTR field. This is the operator-experienced MTTR from landside-observable timestamps. Surfacing the formula footnote makes this provenance visible.

### Recurring-Failure History Block

| Element | Content | Source |
|---|---|---|
| Incident count | "N incidents on this component" — N matches the MTTR lookback window | Aggregated from `recommendation_outbox` + IntentPacket archive |
| Incident list | Compact list of incidents: date / time-of-raise → time-of-clear · duration · resolution status (resolved / ongoing / self-reported-unverified per FR33) | Per-incident from archive |
| Lookback toggle | Affordance to change window: 30 days / 90 days (default) / 180 days / 365 days | Phase 4 may add |
| Click on incident row | Optional: expand to show that specific incident's IntentPackets in a sub-view | Phase 4 may add |

### Last 24h Packet Trace Block

| Element | Content | Source |
|---|---|---|
| Packet log lines | Compact tabular display: timestamp · packet type · severity · brief payload preview | Last N (default 10) IntentPackets for this component in last 24h |
| "view full trace" affordance | Click → opens modal with full 24h+ trace, paginated | Phase 4 confirms modal vs new-tab |
| Filter affordance | Filter trace by packet type (ALARM_RAISED / ALARM_CLEARED / HEALTH_CHECK / etc.) | Phase 4 may add |

### Related Kondukt Recommendation Block

This block appears **only if** there's a Kondukt recommendation related to this component (e.g. the recurring-failure pattern from scenario 11.4).

| Element | Content | Source |
|---|---|---|
| Recommendation summary | One-line description of the related Kondukt emission | Recommendation Envelope (FR29) |
| Recommendation triplet ID | The `recommendation_id` field | FR29 |
| "open in Replay archive" affordance | Click → opens scenario 05.7 archive surface in new tab with this triplet pre-loaded | Routes to Renate's Replay surface |
| Validation Receipt status | Three-check inline (schema / citation-trace / advisory-language) | Per the triplet's Validation Receipt |

### Vendor Portal Deep-Link Block

| Element | Content | Source |
|---|---|---|
| Vendor portal name + button | "Open Stadler portal for this component ↗" (or Siemens / Knorr-Bremse / etc.) | Per-vendor URL template + this component's MAC |
| URL preview | Visible URL with MAC pre-filled, for transparency | Computed from URL template + MAC |
| Click | Opens vendor portal in new tab; URL is pre-filled with MAC; Roland authenticates separately at vendor portal (out of harness scope) | External link |
| Per-vendor coverage | Working assumption: Stadler / Siemens / Knorr-Bremse have templated URLs. Other vendors: best-effort. | Per-vendor configuration |
| Disabled state | If vendor doesn't have a templated URL or component vendor unknown | Block shows "No vendor portal configured for this component" |

### Reveal Affordance on Primary Row

When secondary disclosure is revealed for a component, the "↓ click for diagnostic depth" affordance on the primary row changes to "↑ click to collapse diagnostic depth". Click toggles state.

## Behaviour Notes

- **Secondary disclosure is inline.** Not a modal. Not a side panel. Stays in flow below the primary row. Roland can see multiple secondary disclosures simultaneously on a card if he reveals multiple component rows.
- **Operator interaction logged per FR32.** Every secondary disclosure reveal records `operator_action` with `action_type='card_secondary_disclosed'`, component_id, train_id, IdP-bound identity, timestamp. This is a critical engagement signal — operators using secondary disclosure are using Kanban as the depth surface, not as a queue.
- **Vendor portal deep-link click logged per FR32.** Records `action_type='vendor_portal_deep_link_clicked'`, vendor_name, component_id, MAC, train_id, IdP.
- **Replay archive deep-link click logged.** Records `action_type='replay_archive_opened_from_kanban'`, recommendation_id, train_id, IdP.
- **MTTR formula transparency.** The inline footnote ("t_close − t_open") makes the harness's MTTR ownership claim visible per FR4 + design log "MTTR sourcing reframed" entry. No hidden computation.
- **Multi-component secondary disclosure.** Roland can have multiple component rows secondary-disclosed within a single train's expanded card. Lane scrolls internally if extent exceeds viewport.
- **Real-time updates while revealed.** New IntentPackets for the component update the active-incident MTTR live; new packets append to the trace; recurring-failure history static (only updates when an incident closes).

## Open Questions

- **MTTR lookback window default** — 90 days working assumption. Some operators may want shorter (30-day rolling) or longer (year-to-date). Phase 4 confirms.
- **Recurring-failure incident click expansion** — should clicking an individual incident in the history list reveal that specific incident's IntentPackets? Working assumption: not in MVP; Phase 4 may add as drill-down. The "view full trace" affordance handles current incident; historical incidents can be opened via deep-link to archive.
- **Packet trace pagination** — modal with paginated view for full 24h+ trace? Or sidebar? Or new tab? Working assumption: modal (preserves Kanban context). Phase 4 confirms.
- **Vendor portal URL template management** — where do per-vendor URL templates live? Working assumption: configuration table per ÖBB deployment; ÖBB IT Security signs off on allowed templates (overlap with allowlisted egress proxy domains). Phase 4 confirms.
- **Vendor portal authentication** — Roland authenticates at vendor portal separately (cookies, SSO, etc.). Harness has no role here. Some vendor portals may use OAuth — possibly future Kondukt-mediated fetch. Out of MVP.
- **Stale data threshold for secondary disclosure** — primary disclosure (12.3) has per-component staleness; should secondary disclosure inherit or have its own threshold? Working assumption: inherit per-component staleness; if primary is stale, secondary shows the same staleness indicator prominently.
- **"View full trace" target** — modal vs new tab. Modal keeps Kanban context but blocks; new tab loses context but allows long inspection. Working assumption: modal. Phase 4 confirms.
