## Overview

**Page Purpose:** Roland accepts a Kondukt-proposed lane move with a single click. The card animates to the destination lane; the operator action is recorded against his IdP identity; the move + provenance triplet is written to the Provenance Replay archive. This is the FR9 implementation surface — the moment a Kondukt recommendation becomes an operator decision and a recommendation→action chain row in the shadow audit trail.

This is also the surface where Roland's per-view acceptance rate (FR63 criterion 1) is measured — a critical Phase 2 trust criterion. Acceptance must feel friction-free; the cost of the decision is in the read of the Kondukt proposal banner, not in the click that commits it.

**User Situation:** Roland has read the Kondukt-proposed move on a card ("Move to Investigating — neglect-risk HIGH"), the Validation Receipt shows three green checks (schema / citation-trace / advisory-language). He agrees with the proposal. He clicks "Accept".

**Success Criteria:**
- Single click commits the move; no confirmation dialog (no deliberate-confirmation gesture in MVP per design log 2026-05-25)
- Card animates to destination lane within 400ms perceived
- Operator action recorded in `operator_action` table with `action_type='kanban_lane_move'`, `recommendation_id`, IdP identity, timestamp
- Provenance Replay triplet updated with Roland's action — Envelope + Provenance Bundle + Validation Receipt all reference the accepted move
- Lane card counts update live
- Failure handling: if backend write fails, action surfaces error inline; card returns to original lane; Roland can retry

**Entry Points:**
- Click on "Accept" button on a card with a Kondukt proposal (12.1)
- Keyboard: Enter on focused Accept button
- Equivalent to drag-and-drop manual move when the dropped lane matches Kondukt's proposed lane (recorded slightly differently — `action_type='kanban_lane_move_via_drag_matching_proposal'`)

**Exit Points:**
- Move committed → return to 12.1 with card relocated; subtle confirmation toast
- Failure → error inline on card; Roland retries; no exit until resolved or abandoned

## Layout Structure

Not a standalone page — an action on a Kanban card. The action surface is the card itself; the result is a state transition (card moves to new lane).

```
Before click:
┌─────────────────────────────────┐
│ ▒ 4736-101  CCTV  (Train detail)│
│ Kondukt proposes:               │
│  ↗ Move to Investigating        │
│ [Accept]  [Modify]  [Reject ▼]  │
└─────────────────────────────────┘

Click event captured → button gets focus ring + 50ms press-state animation

Backend write (synchronous, optimistic UI):
  POST /api/kanban/cards/{card_id}/lane-move
  body: {
    target_lane: "investigating",
    recommendation_id: "rcmnd_2026_05_25_06_47_a7c…",
    operator_action_type: "kanban_lane_move",
    accepted: true
  }

Backend response (<300ms typical):
  201 Created
  body: {
    action_id: "act_2026_05_25_…",
    new_lane: "investigating",
    audit_logged: true,
    provenance_updated: true
  }

UI animation (400ms):
  Card lifts (subtle z-elevation)
  Card animates horizontally to Investigating lane
  Card slots into position (newest-first within lane sort)
  Original lane card-count decrements; destination lane card-count increments

Confirmation:
  Toast (3 seconds, dismissible): "Card moved to Investigating. Action logged."
```

## Components

### Accept Button (Kanban card)

| Element | Behaviour |
|---|---|
| Default state | Primary visual emphasis (filled background, contrast text); icon + "Accept" label |
| Hover | Subtle elevation; cursor: pointer |
| Focus | Visible focus ring; keyboard reachable via Tab |
| Click / Enter | Captures the action; press-state animation (50ms); triggers backend write |
| Disabled state | Only when Kondukt proposal is LOW-confidence (per FR27); button shows `aria-disabled="true"` + tooltip routing to Modify path (sub-page 12.5) |
| Loading state | If backend takes >500ms, button shows compact spinner; subsequent clicks are debounced |

### Backend Write

| Field | Source |
|---|---|
| `card_id` | Kanban card identifier (typically derived from `train_id` + most-severe-active-fault context) |
| `target_lane` | Kondukt's proposed lane |
| `recommendation_id` | The triplet ID of the proposal (FR29 Envelope field) |
| `operator_action_type` | `kanban_lane_move` |
| `accepted` | `true` (this sub-page's path); `false` reserved for reject path |
| `operator_id` | IdP-bound identity from session |
| `timestamp` | Server-side; not client-supplied |
| `modification_distance` | `0.0` for acceptance (Roland accepted unmodified); see 12.5 for modification path |
| `from_lane` | Source lane (for audit trail) |

### Animation Sequence

| Phase | Duration | Visual |
|---|---|---|
| Press-state | 50ms | Button slight scale-down |
| Card lift | 100ms | Subtle z-elevation + shadow |
| Card horizontal travel | 250ms | Ease-in-out from source lane to destination lane; cards in destination lane shuffle to make space |
| Card settle | 50ms | Card lands in position; lane card-count updates |
| Toast appear | After settle | Toast slides in from bottom-right; 3s auto-dismiss |

Total perceived latency target: <400ms from click to settle.

### Confirmation Toast

| Element | Content |
|---|---|
| Message | "Card moved to Investigating. Action logged." |
| Icon | ✓ green check |
| Auto-dismiss | 3 seconds; user can dismiss earlier |
| Action affordance | None (no undo in MVP) |
| Live region announcement | Screen reader: "Card for train 4736-101 moved to Investigating lane. Action logged." |

### Operator Action Recording

Every accept produces:

1. **`operator_action` row** — append-only PostgreSQL table:
   - `action_id` (UUID)
   - `operator_id` (IdP)
   - `recommendation_id` (FK to recommendation table)
   - `action_type` = `kanban_lane_move`
   - `from_lane` / `to_lane`
   - `accepted_unmodified` = `true`
   - `modification_distance` = `0.0`
   - `timestamp`

2. **Recommendation triplet update** — the original triplet's Validation Receipt gets an `operator_disposition` field set to `accepted` (per FR32). The triplet itself remains immutable; the disposition is a separate append-only relationship.

3. **Per-view acceptance metric update** — feeds the seven-criteria Phase 2 composite (FR63 criterion 1).

## State Variants

| State | Trigger | Visual |
|---|---|---|
| Default | Card with Kondukt proposal at HIGH or MEDIUM confidence | Accept button enabled, primary emphasis |
| LOW-confidence | Card with Kondukt proposal at LOW confidence | Accept button disabled; route to 12.5 Modify path |
| Click → loading | Backend write in progress | Button shows spinner; subsequent clicks debounced |
| Success | Backend response 201 | Animation sequence triggers; toast appears |
| Backend error | 500 / network failure | Inline error on card: "Action failed — please retry. Error: {short_message}"; Accept button reverts to enabled state; original card remains in source lane |
| Conflict (rare) | Card was moved by another mechanism (manual drag from another tab) between Roland's load and click | 409 Conflict response; inline message: "Card was moved separately. Refreshing view…"; auto-refresh of card state |
| Kondukt provider degraded between view-load and click | FR20 — provider 429 since the proposal was rendered | Proposal still has a valid Validation Receipt (it was emitted earlier); acceptance proceeds normally — the provider's current state does not invalidate prior emissions |

## Behaviour Notes

- **No confirmation dialog.** Single click commits. This is a design-system-level decision (deliberate-confirmation gestures descoped from MVP 2026-05-25). The cost of the decision is in the read of the proposal banner; the click is friction-free.
- **No undo in MVP.** Once committed, the lane move is permanent (the move itself is reversible — Roland can drag the card back — but each move is its own audit row; there's no "undo last action" affordance). Reversibility is via subsequent operator action, not via UI undo. Phase 4 may add undo affordance if Roland reports friction.
- **Acceptance signal feeds the seven-criteria composite.** Every Accept contributes to the per-view acceptance rate (FR63 criterion 1). Sustained sub-70% per-view acceptance is a calibration trip-wire.
- **Drag-and-drop matching proposal** is recorded as `kanban_lane_move_via_drag_matching_proposal` — semantically equivalent to Accept but tracked separately so we can analyse interaction preference during pilot (do operators prefer button or drag?).
- **Telemetry:** event `kanban.lane_move_accepted` with `{card_id, recommendation_id, from_lane, to_lane, time_from_proposal_render_to_accept_ms, time_card_visible_in_viewport_ms}`. The dwell-time-on-viewport signal helps analyse whether Roland read the proposal or rubber-stamped it.

- [ ] LOW-confidence proposals route to 12.5 instead (Accept disabled at this surface)

## Open Questions

- **Undo affordance** — should Phase 4 consider an undo within N seconds after Accept? Working assumption: no undo in MVP; lane moves are reversible via drag, audit row preserved. Phase 4 may revisit.
- **Toast duration** — 3 seconds working assumption. Too short = Roland may miss it; too long = visual noise. Phase 4 confirms.
- **Drag-and-drop vs button preference analysis** — Phase 4 should analyse whether operators prefer one interaction. May inform whether to deprecate one in future.
- **Backend write failure UX** — inline error retains the card in source lane. Should it offer a "save offline / retry later" option in case of sustained network failure? Working assumption: no — degraded mode is detected at FR20 banner level; individual lane-move retries happen at user discretion.
- **Multi-card batch accept** — should Roland be able to accept multiple Kondukt proposals at once? Working assumption: no in MVP; each card is a deliberate read + action. Phase 4 may add if Roland reports friction at scale.
