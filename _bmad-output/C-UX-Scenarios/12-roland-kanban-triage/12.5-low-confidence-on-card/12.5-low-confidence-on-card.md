## Overview

**Page Purpose:** When Kondukt's lane-move proposal carries LOW confidence, the LOW-Confidence DISABLE rule (party-mode lock 2026-05-25 + design system spec) applies: the **Accept button is non-interactive** (`aria-disabled="true"`), and **Modify is the only execution path**. The Modify flow opens an inline justification panel with the cursor pre-placed in the justification field (forcing a moment of operator attention). On commit, `modification_distance` is captured (lexical / semantic diff between Kondukt's draft and Roland's committed text).

This sub-page is the LOW-confidence variant of the same card surface 12.1 + 12.2 describe — same train, same proposal structure, different commit path.

**Why this matters:** Phase 2 trust evaluation depends on distinguishing "Roland accepted Kondukt's recommendation" from "Roland committed a modified version of what Kondukt suggested." Modification distance is the signal that turns LOW-confidence emissions from rubber-stamps into operator-validated decisions. Without this, the seven-criteria Phase 2 composite (FR63 criterion 1 — per-view acceptance) is noisy.

**User Situation:** Roland is on a Kanban card where Kondukt's lane-move proposal has LOW confidence (typically: insufficient citation depth, only one IntentPacket supporting the move, or sparse historical context). The card visually distinguishes itself — outline-only confidence pill, slate-grey severity rail desaturation per design system LOW-Confidence DISABLE rule.

**Success Criteria:**
- Accept button visibly disabled (`aria-disabled="true"` + tooltip explaining the LOW-confidence policy)
- Modify is the primary affordance — visually emphasised
- Click on Modify opens inline justification panel with cursor pre-placed in justification field (no extra click needed to begin typing)
- On commit, `modification_distance` computed and recorded
- Reject path available (with reason)
- Operator action recorded with full LOW-confidence context

**Entry Points:**
- Card in 12.1 with Kondukt proposal at LOW confidence
- All paths from Live Board / Kanban that lead to such a card

**Exit Points:**
- Modify committed → card lane move applied; modification_distance recorded; return to [12.1-lanes-view](../12.1-lanes-view/12.1-lanes-view.md) with card relocated
- Reject committed (with reason) → card stays in current lane; rejection logged; return to 12.1
- Cancel modification → justification panel closes; card returns to LOW-confidence-default state

## Components

### LOW-Confidence Visual Treatment (per design system)

| Element | LOW state |
|---|---|
| Severity-band swatch (left edge) | Desaturated to slate-grey (chroma drains out) — visually distinct from active severity colours |
| Confidence pill | Outline-only (dashed 1.5px border); "LOW" small-caps with tracking-wide |
| Justification block | `data-vouched="false"` — desaturated; prefixed "Unverified context:"; no inline confidence chip |
| Validator indicator | The 3-check Validation Receipt indicator may show ⚠ warning on the weakest validator (e.g. ⚠ schema with caveat, ✓ citation but with note "only 1 packet") |

### Accept Button (DISABLED state)

| Property | Value |
|---|---|
| `aria-disabled` | `"true"` |
| Visual state | Greyed out; cursor: not-allowed on hover |
| Tooltip on hover | "Platform declines to vouch for low-confidence recommendations — use Modify to record your action." |
| Click behaviour | No-op (event captured but no action taken); brief shake animation as feedback |
| Keyboard | Skipped in Tab order (or focusable with `aria-disabled` announcement) |

### Modify Button (PRIMARY on LOW)

| Property | Value |
|---|---|
| Visual emphasis | Primary fill (where Accept would have been); high contrast |
| Click | Reveals inline justification panel below the card; cursor automatically pre-placed in justification field |
| Visual differentiation from Modify-on-HIGH | Modify-on-LOW must visibly differ from Modify-on-HIGH (per design system lock — the same affordance + different intent collapses under time pressure into rubber-stamping). Treatment: outline weight + accent colour on the justification field. |

### Modify-Inline Panel

| Element | Behaviour |
|---|---|
| Header | "Modifying Kondukt's proposal" — sets context |
| Kondukt's draft | Read-only display of the original proposal text |
| Justification field | Textarea, multiline; cursor pre-placed (auto-focus on render with `selectionStart = 0`); placeholder: "Why are you committing this modification?"; required field |
| Destination lane dropdown | Pre-selected to Kondukt's proposed destination; Roland can change to any of the four lanes |
| Commit button | Disabled until justification field is non-empty (min 5 chars; configurable) |
| Cancel button | Closes panel; returns card to LOW-default state |
| Character count | Optional small indicator: "{n} characters" — soft minimum 20 chars suggested (not enforced) |

### Modification Distance Capture

On commit:
- **Lexical distance:** Levenshtein distance between Kondukt's draft text and the operator's committed justification (normalised by length)
- **Semantic distance:** cosine similarity between embeddings of the two texts (Kondukt's draft vs Roland's text). Computed via the harness's pgvector embedding model.
- Both stored as `modification_distance.lexical` and `modification_distance.semantic` in the `operator_action` row

The capture is **required** (FR28 + design log pre-mortem note 2026-05-25). Without modification_distance, Phase 2 trust dataset becomes noise — a 1.8s rubber-stamp can't be distinguished from a substantive rewrite.

### Reject Button + Reason

| Element | Behaviour |
|---|---|
| Reject with reason dropdown | Options: "Already triaged" / "Not actionable today" / "Disagree with Kondukt" / "Information insufficient" / "Other (text)" |
| Reject commits | Card stays in source lane; rejection logged with `action_type='lane_move_rejected'`, recommendation_id, reason, IdP identity |
| No justification text needed for predefined reasons | Other-with-text option requires text input |

### Operator Action Recording (LOW-Confidence Variant)

Every Modify commit produces:

| Field | Value |
|---|---|
| `action_id` | UUID |
| `operator_id` | IdP |
| `recommendation_id` | FK to original LOW-confidence proposal |
| `action_type` | `kanban_lane_move_modified` (distinct from `kanban_lane_move` to keep LOW-confidence path visible in audit) |
| `accepted_unmodified` | `false` |
| `modification_distance.lexical` | computed |
| `modification_distance.semantic` | computed |
| `modification_text` | Roland's committed justification |
| `destination_lane` | Roland's selected lane (may differ from Kondukt's) |
| `confidence_at_proposal_time` | `LOW` |
| `timestamp` | server-side |

## State Variants

| State | Trigger | Visual |
|---|---|---|
| Default (LOW-confidence card displayed) | Card with Kondukt LOW-confidence proposal | LOW visual treatment as above; Accept disabled, Modify primary, Reject available |
| Hover Accept (disabled) | Mouse over disabled Accept button | Tooltip appears; cursor: not-allowed |

| Modify cancelled | Click Cancel | Panel closes; card returns to LOW-default state; no action recorded |
| Reject panel | Click Reject dropdown | Reason picker appears; on selection (and optional text), commits rejection |
| Backend error on Modify commit | Network / 500 | Inline error in panel; Commit button re-enabled; Roland can retry |
| LOW-rate trip-wire | (Observed across pilot, not per-card) | When LOW-rate exceeds 15% sustained over any 72h window, harness ops dashboard surfaces a calibration alert (Roland sees no UI change here, but Nomad ops + Renate audit are flagged) |

## Behaviour Notes

- **No deliberate-confirmation gesture.** The Modify path commits on single button click — same as the HIGH/MEDIUM Accept path (12.2). The friction is in the **justification typing**, not in the commit gesture. This is intentional per design log 2026-05-25 (deliberate-confirmation gestures descoped from MVP).
- **Cursor pre-placement is the forcing function.** When Modify is clicked, the textarea receives focus with cursor at position 0. Roland *must* type at least the minimum character count to commit. This is a small ergonomic detail with large signal-quality consequence (per design system pre-mortem note).
- **Modify-on-LOW differentiation from Modify-on-HIGH.** Design system locks: Modify on LOW must visibly differ from Modify on HIGH (otherwise the same affordance + different intent collapses under time pressure into rubber-stamping). The differentiation: outline weight + accent colour on the justification field; LOW shows the slate-grey desaturation that bleeds into the modification panel.
- **Modification distance computation.** Lexical distance via Levenshtein is cheap; semantic distance via embedding requires a backend call to pgvector. Both stored. Phase 4 may add a third measure (e.g. operator-edit-time as proxy for thoughtfulness).
- **LOW-rate as operational metric.** LOW-rate tracking is at the ops-dashboard level, not visible to Roland on individual cards. >15% sustained over 72h triggers calibration review. >30% sustained means the forced-Modify friction stops being signal and starts being theatre.

- [ ] Cancel returns card to LOW-default state without recording action
- [ ] Backend errors surface inline with retry

## Open Questions

- **Minimum justification character count** — 5 chars working assumption. Too low = trivial bypass; too high = friction. Phase 4 confirms with pilot data.
- **Semantic distance computation cost** — embedding API call per commit adds latency. Working assumption: async — commit succeeds immediately, embedding computed in background, stored when ready. If embedding API fails, lexical distance stored alone. Phase 4 confirms.
- **Re-Modify after Commit** — if Roland wants to revise his modification after committing, what's the path? Working assumption: a new operator_action row (new modification), original retained in audit. No edit-in-place.
- **Modify-on-LOW differentiation visual treatment** — design system locks the *concept* (must differ visually); specific treatment (outline weight + accent) is a working assumption. Phase 4 confirms with mockups + usability testing.
- **LOW-rate trip-wire alerting** — who sees the alert? Working assumption: Nomad ops dashboard + Renate audit committee summary. Phase 4 confirms whether Roland sees it (probably not — it's a calibration signal, not an operator signal).
- **Reject reason taxonomy** — 4 predefined + Other-with-text working assumption. Phase 4 may refine after pilot use.
- **Cursor pre-placement consistency** — every Modify-on-LOW must auto-focus the justification field on render. Testing with screen-readers + keyboard-only users in Phase 4 to confirm accessibility.
