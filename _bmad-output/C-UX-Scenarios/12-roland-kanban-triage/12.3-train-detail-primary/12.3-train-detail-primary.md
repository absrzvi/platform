## Overview

**Page Purpose:** When Roland expands a Kanban card (via chevron click, title click, or auto-expansion from Live Board click-through scenario 11.5), the card reveals **primary disclosure**: per-component diagnostic depth for every affected subsystem on the train. Each component row shows the operator-friendly subsystem name + VLAN ID + IP address + MAC address + switch port + firmware version + time-since-fault. This is the FR10 primary-disclosure layer — the depth that replaces vendor-portal-lookup + Power-BI-historical reconciliation friction.

**No VLAN IDs appear on Live Board (FR2);** they appear here, on the Kanban detail card, where Roland goes for diagnostic depth. The primary disclosure is the depth Roland needs to **identify the component** without further click. Secondary disclosure (12.4) is what he needs to **investigate** it.

**User Situation:** Roland either:
1. Entered Kanban via Live Board click-through (scenario 11.5) — card is auto-expanded at primary disclosure; 2-second highlight visible
2. Clicked a card title or expand chevron from 12.1 — card expands to primary disclosure

Either way, this sub-page describes the rendered state.

**Success Criteria:**
- Per-component rows visible: subsystem name + VLAN + IP + MAC + switch port + firmware + time-since-fault — all for every component with an active fault on this train
- Rows sortable / filterable by severity, by time-since-fault, by subsystem class
- Each row is clickable to reveal secondary disclosure (12.4)
- Component rows clearly distinguish active-fault components from healthy components (latter only shown if Roland toggles "show all components")
- Card visual extent expands within the lane; lane scrolls if necessary
- Auto-expansion (from Live Board click-through) renders within 100ms of card render; 2-second highlight clearly visible

**Entry Points:**
- Card title click on 12.1
- Expand chevron click on 12.1
- Auto-expansion from URL param `?train={id}&detail=expanded` (Live Board click-through 11.5)
- Keyboard: Enter on focused card title

**Exit Points:**
- Click on a component row → [12.4-train-detail-secondary](../12.4-train-detail-secondary/12.4-train-detail-secondary.md) (secondary disclosure for that component)
- Click expand chevron again → card collapses back to summary (12.1 state for that card)
- Click "Show all components (including healthy)" toggle → expand row list to all subsystems
- Navigate to Live Board / Kondukt → exits Kanban entirely

## Components

### Card Header (always visible on expanded card)

Same as the collapsed card header (12.1) but with one addition:
- **(N components affected)** appended to the dominant subsystem indicator — e.g. "CCTV (3 components affected)" if multiple components in different subsystems are affected on this train.

### Component Detail Block

The primary-disclosure block — visible only when card is expanded.

For each affected component, render one **Component Row**:

| Field | Display | Source |
|---|---|---|
| Severity-band swatch | Left edge of row, 4px coloured strip | Per-component severity from IntentPacket |
| Subsystem name + component label | e.g. "CCTV · Überw. Kamera A7" — bold | IP allocation reference (operator-friendly names) |
| VLAN | "VLAN 5" | IntentPacket / IP allocation reference |
| IP address | "IP 172.18.192.138" | IntentPacket payload |
| MAC address | "MAC AA:BB:CC:11:22:33" | IntentPacket payload |
| Switch port | "port e1-0" | IP allocation reference |
| Firmware version | "firmware v2.3.1" | IntentPacket payload (versions vary per subsystem) |
| Time-since-fault | "fault active 4h 23m" (relative format) | Computed from earliest active-fault IntentPacket arrival |
| Reveal affordance | "↓ click for diagnostic depth" or expand chevron | Click → 12.4 |

**Row visual treatment:**
- Severity-band swatch matches the design system severity colour (CRITICAL red / HIGH orange / MEDIUM amber / LOW yellow)
- Subsystem name in bold; metadata (VLAN/IP/MAC/port/firmware) in slightly smaller text
- Time-since-fault as a small badge or label
- Click target: entire row (cursor pointer)

**Row sort order (default):**
- Primary: severity (worst first)
- Secondary: time-since-fault (oldest first)
- Tertiary: subsystem class alphabetical

**Hidden by default:** components without active faults. Roland can toggle "Show all components (including healthy)" to see the full per-train subsystem inventory. Useful when investigating cross-component correlations (e.g. "is the healthy switch on the same port as the failing camera?"). Toggle state persists for the session.

### Component Row Variants

| Variant | Trigger | Visual |
|---|---|---|
| Active fault (default) | Component has an active fault IntentPacket | Severity-band swatch coloured; row at full opacity |
| Healthy (only when toggle is on) | Component has no active fault | Grey severity-band swatch (no fault colour); row at 60% opacity; "no active fault" label instead of time-since-fault |
| Stale data (per FR59) | Component's last IntentPacket exceeds staleness threshold | Severity-band swatch with stale-data badge; row at 80% opacity; "stale — last update {timestamp}" label |
| Multi-fault | Component has multiple active faults | Severity-band swatch shows worst severity; faults listed inline as comma-separated brief labels |

### "Show all components" toggle

| Element | Behaviour |
|---|---|
| Default state | Off — only active-fault components shown |
| Click | Toggles state; reveals or hides healthy components |
| State persistence | Per-user, per-card; toggle remembered across sessions |

### Collapse Affordance

Bottom of expanded card. Click → card collapses back to 12.1 summary state for that card.

## State Variants

| State | Trigger | Visual |
|---|---|---|
| Default expanded | User clicked card title or chevron | All active-fault component rows visible; sorted by severity |
| Auto-expanded from Live Board click-through | URL param `?detail=expanded` on entry | Card auto-expanded on render; 2-second highlight (border accent or background tint); page scroll positions card centre-of-view |
| Single component affected | Train has only 1 active-fault component | Component row shown alone; no toggle needed |
| Many components affected (>10) | Edge case — train has many simultaneous faults | All rows visible; lane scrolls internally if necessary; "X components affected" header includes the count |
| All healthy (toggle on, train healthy) | Edge case — Roland expanded a Cleared lane card | All component rows shown as healthy; no fault details; "No active faults on this train" message at top of block |
| Stale data on some components | Subset of components have stale IntentPackets | Those rows show stale badge + relative timestamp |
| Card transition in progress | Card moving between lanes (12.2 happening on this card) | Expanded card animates with the move; expansion state preserved |

## Behaviour Notes

- **Card expansion is NOT a route change.** Expansion is an inline UI state change. URL remains `/kanban` (or `/kanban?train={id}&detail=expanded` from Live Board entry — the URL reflects the entry state but doesn't update as Roland expands additional cards). Phase 4 confirms whether multi-card expansion should update URL.
- **Multi-card expansion:** Roland can have multiple cards expanded simultaneously. Each card's primary disclosure is independent. No exclusivity. Lane height grows accordingly.
- **Real-time updates while expanded:** if a new IntentPacket arrives for a component while Roland has the card expanded, the row updates in place (severity, time-since-fault, firmware version on update). Brief flash to highlight the update.
- **Component row click reveals 12.4 secondary disclosure inline below the row.** Not a modal. Multiple component rows can have secondary disclosure revealed simultaneously.
- **Keyboard navigation:** Tab through component rows; Enter opens secondary disclosure (12.4); arrow keys navigate rows within a card.
- **Performance:** card expansion render <100ms; component row data lazy-fetched on viewport visibility (IntersectionObserver pattern).

## Open Questions

- **URL update on multi-card expansion** — should additional expansions update URL? Working assumption: no, URL reflects the entry state only. Phase 4 confirms.
- **Component row icons** — should each subsystem class have a small icon next to the name (camera for CCTV, screen for PIS, etc.)? Working assumption: text-only for MVP; icons as Phase 4 polish.
- **Stale-data badge threshold** — what's the staleness threshold for a component-row stale badge? Per-train FR1 has a `data_freshness_ts` threshold (working assumption ~3 minutes); per-component may need a separate threshold. Phase 4 confirms.
- **Firmware version display format** — semantic version (v2.3.1) is the working assumption. Some subsystems may have non-semver (e.g. build-date strings). Phase 4 confirms display rules per subsystem class.
- **MAC address display** — full 6-byte hex with colons working assumption. Some operators may want truncated display ("AA:BB:CC:…:33"). Phase 4 confirms.
- **Click target for row** — entire row clickable, or only a specific affordance (chevron / "click for diagnostic depth" label)? Working assumption: entire row clickable (larger target, accessibility-friendly); affordance label is visual indicator only.
- **"Show all components" toggle scope** — per-card or global preference? Working assumption: per-card (different trains may need different default views). Phase 4 confirms.
