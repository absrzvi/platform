## Overview

**Page Purpose:** Roland's first view of the fleet on session open. The Live Board is glanceable: a moving map of every train (in-service + depot) with per-subsystem health legible in <60 seconds, no expansion required. Pivot state restored per-user from the previous session (FR3). This is the surface that earns the "harness is THE tab" claim — Roland gets his morning situational awareness here without opening Power BI / Nomad Monitoring / vendor portals.

**User Situation:** 06:30 start of shift. Roland has just authenticated (IdP redirect). BYOK key already configured (from scenario 15 setup, first-time only). His last-shift pivot state was subsystem-pivot (FR3 — per-user state restore).

**Success Criteria:**
- Roland reads fleet state in <60 seconds — no expansion needed
- Per-subsystem health legible at a glance
- Stale-telemetry indicator visible per-train where `data_freshness_ts` exceeds threshold (FR1)
- Kondukt annotations (if any) surface inline without dominating the layout
- Last pivot state restored correctly

**Entry Points:**
- Authenticated session open (default landing surface) — primary entry
- Navigation from Kanban or Kondukt views (top-nav "Live Board" link)
- Browser back from Kanban detail card (state restored)

**Exit Points:**
- Pivot toggle (subsystem → train or vice versa) → [11.2](../11.2-subsystem-pivot/11.2-subsystem-pivot.md) or [11.3](../11.3-train-pivot/11.3-train-pivot.md)
- Click on Kondukt annotation banner → [11.4](../11.4-kondukt-annotation/11.4-kondukt-annotation.md)
- Click on a train (map marker or row) → [11.5](../11.5-click-through-to-kanban/11.5-click-through-to-kanban.md) → scenario 12 entry
- Navigate to Kanban (top-nav) → scenario 12
- Navigate to Kondukt (top-nav) → scenario 13 entry

## Layout Structure

Single-page, full-viewport layout. No modal, no sidebar overlay on default state.

```
+--------------------------------------------------------------------+
| Top Chrome: [logo] Live Board | Kanban | Kondukt    [Roland · IdP] |
+--------------------------------------------------------------------+
|                                                                    |
| Fleet Overview Bar (top edge — sticky)                             |
| 47 in-service · 12 depot · 18 active faults · pivot: subsystem ▼   |
|                                                                    |
+--------------------------------------------------------------------+
|                                                                    |
|                                                                    |
|           [Moving Map Area]                                        |
|           - ÖBB rail network overlay                               |
|           - Train markers (severity-banded)                        |
|           - Subsystem-pivot OR train-pivot layout                  |
|           - Kondukt annotation banners (top-right of subsystem     |
|             columns or floating per-train)                         |
|           - Cellular dead-zone overlay (toggleable; default off)  |
|                                                                    |
|                                                                    |
+--------------------------------------------------------------------+
| Status Strip (bottom — sticky)                                    |
| Mapping health: ✓ · IntentPacket lag: 1.4s avg · Kondukt: HEALTHY |
+--------------------------------------------------------------------+
```

**Region semantics:**
- **Top chrome (24px height):** persistent across all harness views. Logo, view nav, identity menu. ÖBB-brand colours; minimal.
- **Fleet Overview Bar (40px sticky):** quick stats + pivot toggle. Sticks to top of map area on scroll (unlikely to scroll, but possible at <1080 vertical).
- **Map area (fill remainder):** the primary surface. Vector tile background + ÖBB rail network overlay + train markers + Kondukt annotation banners.
- **Status Strip (32px sticky bottom):** harness ops health — mapping health (FR41), IntentPacket ingest lag (FR38 staleness indicator at fleet aggregate), Kondukt provider status (FR20). Always visible.

|---|---|---|

| Pivot toggle | "Pivot: [subsystem ▼]" | Click → dropdown with two options: "By subsystem" (default for first-run) / "By train". Selection persists per-user (FR3). |

### Moving Map

| Element | Behaviour |
|---|---|
| Base tile layer | Vector tile (self-hosted OSM-based); zoom levels 6–14 functional; default 8 (Austria-scale) |
| ÖBB rail network overlay | Lines + station markers from ÖBB infrastructure GIS data; minimal styling so trains visually dominate |
| Train markers | Circle markers (in-service) or square markers (depot); size scales with severity (higher severity → larger); colour = severity band (per design system: CRITICAL red / HIGH orange / MEDIUM amber / LOW yellow / NORMAL grey) |
| Marker hover | Tooltip: train ID, position, top-fault summary one-liner, time-since-fault. No expansion. |
| Marker click | Redirect to that train's Kanban detail card ([11.5](../11.5-click-through-to-kanban/11.5-click-through-to-kanban.md)). |
| Cellular dead-zone overlay | Toggleable layer; default OFF in 11.1; ON in 11.2 if user enabled previously. Shaded regions on the map where >30% of trains have reported stale-telemetry in last 24h. Disambiguates FR7. |

### Subsystem-pivot vs train-pivot rendering (default = subsystem-pivot per first-run policy; user-pivot state restored on subsequent sessions)

In **subsystem-pivot** view (FR3): right edge of map carries an aggregated severity column per subsystem — CCTV / PIS / intercoms / AFZ / audio / Wi-Fi / switches / TCMS / Stadler diagnostic. Each column shows a count of active faults + colour-bar of severity distribution. Click a column → filters map markers to trains with active fault in that subsystem.

In **train-pivot** view: no right-edge subsystem column; instead each train marker carries a small inline subsystem-state ring (or stacked dot pattern) showing per-train subsystem severity. More information density per-train, less per-subsystem aggregate visibility.

This page (11.1) shows whichever view Roland last used. For purposes of detailed spec, sub-pages 11.2 + 11.3 describe each view's specifics.

### Kondukt Annotation Banner (inline)

| Element | Behaviour |
|---|---|
| Position | Top-right of a subsystem column (subsystem-pivot view) OR floating near affected train markers (train-pivot view) |

## State Variants

| State | Trigger | Visual |
|---|---|---|
| Default (loaded, healthy) | Initial load complete; all sub-data present | All zones populated; no banners other than Kondukt annotations |
| Initial loading | First 1–2 seconds after view load | Map skeleton + "Loading fleet state…" centred text; status strip pending |
| No data | IntentPacket ingest failure across the fleet | Banner above status strip: "No fresh fleet data — last IntentPacket received {timestamp}. Mapping health: {state}." Map shows last-known-good train positions with stale indicator on every marker. |
| Train-mapping divergence | FR41 weekly check has flagged a divergence | Status strip mapping-health icon turns red; banner above status strip: "Mapping divergence detected on {train_id} — harness data writes for this train halted. {Link to ops detail.}" Affected train marker greyed out with "DATA HOLD" annotation. |
| Kondukt suppressed/degraded | FR20 — BYOK LLM unreachable | Kondukt annotations cleared from map; status strip "Kondukt: PAUSED — provider 429. Live Board telemetry views unaffected." Kondukt annotation area shows: "Annotations paused — provider unreachable. Previous annotations remain visible in archive." |
| Cellular dead-zone overlay ON | User toggled it in previous session OR from this view (toggle is a sub-control, not detailed on 11.1) | Shaded regions visible on map; stale-telemetry indicators on trains within dead-zone show secondary cause label "stale — geographic" vs default "stale — hardware/network" |
| Pivot-state not yet persisted (first-run for this user) | First-time user, no per-user pivot state in DB | Defaults to subsystem-pivot; saves state on first user-initiated pivot toggle |

## Behaviour Notes

- **Live updates** via SSE on `oebb/{train_id}/intent` stream → harness MQTT consumer → SSE push to client. Map marker position + severity band updates without page refresh. Throttled client-side to max 1 update per train per 2 seconds to prevent flicker.
- **Hover behaviour** on train markers is the only inline expansion permitted on Live Board. Click → redirect; hover → tooltip-only. This enforces the "Live Board stays glanceable" principle.
- **Keyboard navigation:** `Cmd/Ctrl + K` opens Kondukt overlay (quick chat / skill invocation); `Cmd/Ctrl + B` toggles cellular dead-zone overlay; `Cmd/Ctrl + 1/2/3` switches between Live Board / Kanban / Kondukt; arrow keys pan map; `+`/`-` zoom; `f` zooms to filtered-fleet bounds.
- **Performance budget:** initial render <2s; per-update render <100ms; cellular dead-zone overlay toggle <300ms.
- **Per-user pivot state** persists in client localStorage AND server-side `user_preference` table. Server-side is authoritative on cross-device sessions; localStorage is for the current session only.

## Open Questions

- **Marker overlap at low zoom** — when Roland is zoomed to Austria-wide view, train markers overlap. Cluster them (Mapbox-style cluster bubbles)? Or always show individual markers? Working assumption: cluster at zoom <8, individual at ≥8. Phase 4 confirms.
- **Click-through redirect target on map cluster click** — if user clicks a cluster bubble (zoom <8), do we redirect to one train (which?) or expand the cluster? Working assumption: cluster click zooms in one level (no redirect); train marker click (individual) redirects to Kanban.
- **Severity-band aggregation rule for trains with multiple open faults** — highest severity wins is the working assumption. Confirm.
- **Refresh indicator behaviour** when network connection drops — pulsing dot stops, status strip shows "IntentPacket: NO CONNECTION", trains gradually grey out as their `data_freshness_ts` ages. Phase 4 confirms the exact thresholds.
- **Identity menu provider-info display detail** — full provider name + model version? Just "Anthropic" / "OpenAI"? Working assumption: provider + model version + last-success timestamp. Phase 4 confirms.
