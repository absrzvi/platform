## Overview

**Page Purpose:** The transition surface for Live Board → Kanban redirect. When Roland clicks a train marker on Live Board (or a train row in the trains-with-faults panel, or an affected-train row in a Kondukt annotation expansion), the harness navigates to that train's Kanban detail card with the card auto-expanded. This sub-page documents the transition itself — what happens, how browser state is preserved, what the operator sees during the redirect, and how the Kanban entry state is initialised.

This is the FR6 implementation surface. Live Board click → Kanban redirect → card auto-expanded. Live Board does NOT expand inline; depth lives on Kanban.

**User Situation:** Roland has decided to investigate 4736-101 (the top train in the Kondukt-flagged CCTV-A7 cluster). He clicks the train. The harness navigates him to Kanban.

**Success Criteria:**
- Click-to-Kanban navigation perceived <500ms (PRD §10 NFR)
- Kanban entry state: that train's detail card auto-expanded at primary-disclosure level (subsystem name + VLAN + IP + MAC + switch port + firmware + time-since-fault visible)
- Live Board state preserved in browser history — back button returns Roland to his Live Board view + pivot + filters
- Browser URL updated to reflect the new view + train context (`/kanban?train=4736-101&detail=expanded`)
- Operator interaction logged per FR32 — Live Board click → Kanban open is an `operator_action` row with `action_type='live_board_click_through'`

**Entry Points:**
- Train marker click on map (11.1, 11.2, 11.3)
- Train row click in trains-with-faults panel (11.3)
- Affected-train row click in Kondukt annotation expansion (11.4)
- "Investigate first train" CTA in Kondukt annotation (11.4)
- Keyboard: Enter on focused train marker or row

**Exit Points:**
- Loaded Kanban detail card view (scenario 12 — specifically the auto-expanded primary-disclosure state of the train detail card)
- Browser back → returns to Live Board with state restored (pivot, filters, scroll/zoom)

## Layout Structure

Not a layout; a transition. The sub-page documents:

1. **Click moment (Live Board side)** — Roland's mouse click or Enter keypress lands.
2. **Transition state** — Live Board view fades or shifts; URL updates; Kanban view begins loading.
3. **Kanban entry state** — Kanban view rendered; the requested train's detail card auto-expanded at primary-disclosure level.

## Components

### Click Capture

| Element | Behaviour |
|---|---|
| Train marker click (map) | Captures click event; resolves `train_id` from marker data attribute; initiates redirect |
| Train row click (trains-with-faults panel) | Captures row click; resolves `train_id`; initiates redirect |
| Affected-train row click (Kondukt annotation expansion) | Captures row click; resolves `train_id` + emission context; initiates redirect (carrying annotation context to Kanban URL params) |
| Keyboard Enter | Same as click on focused element |
| Modifier-key behaviour | Cmd/Ctrl + click → open in new tab (Kanban view loaded in new tab, Live Board preserved in original); Shift + click → open in new window |

### Transition Animation

| Element | Behaviour |
|---|---|
| Live Board fade-out | View dims to ~60% opacity for ~150ms while Kanban loads |
| Kanban fade-in | View appears at full opacity once render-ready; cross-fade with Live Board |
| Total perceived latency | <500ms target (PRD §10 NFR); ideally <300ms for non-cold caches |
| No spinner | Avoid blocking spinner; cross-fade is the visual cue. If Kanban render exceeds 500ms (rare, only on cold cache), skeleton placeholder visible during render |

### URL State Update

| Element | Behaviour |
|---|---|
| URL before click | `/live-board?pivot=subsystem&filter=cctv` (example) |
| URL after click | `/kanban?train=4736-101&detail=expanded&from=live-board` |
| Browser history | Push entry; back button returns to Live Board URL with full state |
| Deep-link | URL is shareable — anyone with access can paste it and land on 4736-101's expanded Kanban card |

### Kanban Entry State Initialisation

| Element | Behaviour |
|---|---|
| View load | Kanban renders with all lanes visible, sorted by Kondukt's neglect-risk score per lane |
| Card location | The 4736-101 card is located in its lane (whichever lane it's currently in — typically "New" or "Investigating"); page scrolls to position the card centre-of-view |
| Card auto-expand | Card automatically expands to **primary disclosure level** (FR10): subsystem name + VLAN ID + IP + MAC + switch port + firmware + time-since-fault per component |
| Card highlight | Card has a subtle visual highlight (background tint or border accent) for 2 seconds to confirm "this is the one you clicked" |
| Secondary disclosure | NOT auto-expanded; Roland clicks to reveal (per FR10 progressive disclosure) |
| `from=live-board` URL param | Used by Kanban view to add a "Back to Live Board" affordance in the breadcrumb |

### Browser Back Behaviour

| Element | Behaviour |
|---|---|
| Back button | Returns to Live Board with state restored from URL (pivot, filters) + client-side state (zoom, pan, scroll) |
| Live Board state preservation | Pivot: persisted server-side (FR3); filters: session-only client state; zoom/pan/scroll: session-only client state |
| Kondukt annotation expansion state | NOT preserved across navigation — Roland sees Live Board in default state on return. Phase 4 may revisit this; see Open Questions. |

## State Variants

| State | Trigger | Visual |
|---|---|---|
| Default click-through | Train marker click | Transition as above; Kanban renders with primary-disclosure expansion |
| Modifier-key click | Cmd/Ctrl + click | New tab opens to Kanban; original Live Board unchanged |
| Kanban load failure | Network failure or backend error | Error page with retry affordance; Live Board state preserved in browser history for fallback |
| Train no longer exists in fleet | Edge case — train was removed between Live Board render and click | Error page: "Train 4736-101 not found. Mapping may have been updated. [Return to Live Board]" |
| Card already expanded by another mechanism (rare) | If Kanban already had this card expanded for some reason | Behaviour: maintain expansion, scroll-into-view, apply highlight |
| Slow render (>500ms) | Cold cache or heavy fleet | Cross-fade pauses at 60% opacity until Kanban renders; skeleton placeholder visible if >800ms |

## Behaviour Notes

- **Operator action logged:** every click-through records an `operator_action` row with `action_type='live_board_click_through'`, `recommendation_id` if from a Kondukt annotation (FR32), source view (subsystem-pivot vs train-pivot vs annotation-expansion), source train_id. This feeds the per-view acceptance metric (FR63 criterion 1).
- **Performance:** Live Board → Kanban perceived latency is part of the harness UX contract. Slow click-through erodes the "harness is THE tab" claim (Roland F3).
- **No modal**: this is full-page navigation. Modal-overlay alternatives were considered and rejected because Kanban itself is a primary view, not a detail of Live Board.
- **Keyboard accessibility:** Enter on focused train marker triggers click-through; Tab moves focus across markers in z-order; arrow keys pan map.
- **Telemetry events for pilot analytics:** event `live_board.train_clicked` with properties `{train_id, source_view, kondukt_annotation_id?, dwell_time_ms_on_live_board, time_to_kanban_render_ms}`.

## Acceptance Criteria

- [ ] Train marker click on Live Board (any view variant) navigates to Kanban with that train's detail card auto-expanded at primary-disclosure level (FR6, FR10)
- [ ] Perceived latency <500ms (target <300ms warm cache)
- [ ] URL updates to `/kanban?train={id}&detail=expanded&from=live-board`
- [ ] Browser back returns to Live Board with state restored
- [ ] Cmd/Ctrl + click opens Kanban in new tab; Shift + click in new window
- [ ] Operator action logged per FR32 with full context
- [ ] No spinner blocking; cross-fade transition; skeleton only on cold-cache renders
- [ ] Affected-train rows in Kondukt annotation expansion route through the same click-through logic
- [ ] Edge case: train no longer in fleet renders error page with return-to-Live-Board affordance
- [ ] Card auto-expansion respects FR10 progressive disclosure — primary visible, secondary NOT auto-expanded

## Open Questions

- **Should Kondukt annotation expansion state be preserved across navigation?** Currently spec'd as NOT preserved. Argument for preserving: if Roland investigates one train from the annotation and returns to Live Board, he probably wants to see the annotation still open to investigate the next train. Argument against: state preservation across navigation adds client-side complexity. Working assumption: NOT preserved in MVP; Phase 4 revisit if Roland reports friction.
- **Auto-expansion delay** — should the Kanban card expand instantly on render, or with a 200ms delay for the user's eye to settle? Working assumption: instant; Phase 4 confirms with usability testing.
- **Multi-train click-through from annotation expansion** — when Roland is in the Kondukt annotation expansion and clicks the first affected train, the annotation overlay collapses. If he wants to investigate the second affected train, he must re-open the annotation. Alternative: keep annotation overlay open; investigate-multiple flow. Working assumption: simpler one-train-at-a-time MVP; Phase 4 may add multi-investigate.
- **Telemetry storage** — `live_board.train_clicked` events go to PostgreSQL `operator_action` table. Aggregation queries for the per-view acceptance metric are computed on-demand. Phase 4 confirms query performance budget.
- **Card highlight duration** — 2 seconds is the working assumption. Too short = Roland may miss it; too long = visual noise. Phase 4 confirms.
