## Overview

**Page Purpose:** Stefan modifies the RAO's payload in-place. The panel shows: (1) original payload (read-only, struck-through where changed), (2) editable fields (current values), (3) diff summary, (4) a required "reason for modification" field, and (5) the path to executing with modification recorded.

**This is the scenario's most novel surface.** It is *the* place the modification-rate signal is captured. Both the original and the modification persist immutably in the audit row.

**User Situation:** Stefan clicked "Modify" on the RAO card (02.3). He intends to change a payload field (e.g., maintenance slot from Friday 22:00 → Saturday 04:00) and capture his reasoning.

**Success Criteria:**
- Original payload preserved immutably (visible alongside modification, struck-through fields make change obvious)
- Reason-for-modification field is REQUIRED; cannot submit without it
- Diff is shown before submit ("Friday 22:00 → Saturday 04:00")
- Save-with-modification creates a single audit row carrying both states, then routes to the same Mark Executed gesture (01.6) for the actual operator-action recording

**Entry Points:**
- "Modify" button on RAO card (02.3 or 01.5)

**Exit Points:**
- "Mark executed (with modification)" → [01.6-mark-executed](../../01-stefan-crisis-redeployment/01.6-mark-executed/01.6-mark-executed.md) (with mod state carried forward)
- "Cancel modification" → returns to [02.3-rao-card](../02.3-rao-card/02.3-rao-card.md) (no audit-row write)

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Modifying — no changes yet | Stefan entered modify mode; field grid editable but no field changed | Diff summary empty; Mark Executed disabled; Reason field empty | Edit fields, type reason, cancel |
| Modifying — field changed, no reason | Stefan changed a field; reason empty | Diff summary populated; Mark Executed disabled; reason field highlighted as required | Type reason, more field changes, cancel |
| Modifying — ready | At least one field changed AND reason ≥8 chars | Diff summary populated; Mark Executed enabled (primary fill) | Mark executed, more changes, cancel |
| Validation error | Stefan attempts invalid value (e.g., maintenance slot in the past, unit not in fleet) | Inline field error message; Mark Executed disabled until resolved | Fix field |
| Server-side justification regenerating | After Mark Executed is clicked, WHY + STEPS regenerate based on new payload | Brief loading state; Stefan sees the updated justification before the hold-to-record modal opens (Phase 4 open question — confirm sequencing) | Watch |

## Technical Notes

- **Immutable original:** the RAO row in the database is never mutated. On Mark Executed (with mod), a *new* audit row is created with:
  ```
  { original_rao_id, modified_payload, modification_reason, modified_by_idp, modification_at, hold_started_at, ... }
  ```
  The original RAO remains queryable for the recommendation-acceptance-rate dataset; the modification row joins it for modification-rate signal.
- **Server-side justification regeneration:** when Stefan submits Mark Executed (with mod), the server *regenerates* WHY and STEPS to match the new payload (so the next-shift Disponent reading this row sees a coherent justification). The original WHY and STEPS are persisted in the audit row but NOT shown unless explicitly requested.
- **Field-type validation:** depends on payload field. Maintenance slot uses date-time-with-availability validation; unit_id validates against active fleet; rotation_id validates against HAFAS rotation set.
- **Common reasons taxonomy:** seeded list; not user-editable in MVP. ECM Manager sign-off on the taxonomy is a pre-pilot blocker (analogous to Elena's DEFER reason taxonomy in scenario 04).
- **Carry-forward to 01.6:** the modification panel does NOT write the audit row itself. It collects state and passes it to [01.6-mark-executed](../../01-stefan-crisis-redeployment/01.6-mark-executed/01.6-mark-executed.md) where the hold-to-record gesture commits it. This preserves single-gesture single-write contract.
