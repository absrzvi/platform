## Overview

**Page Purpose:** Stefan's triage view with execution log showing the just-completed RAO with **modification visibly captured**. Same surface as 01.7, but the log entry uses the "EXECUTED W/ MOD" badge variant, makes the original visible via expand, and surfaces the reason for modification on the row.

This is the row Renate's audit committee will read most closely. It is also where modification-rate signal manifests visibly during Stefan's shift.

### Sub-structure (extends 01.7 `execution-log-entry-{id}`)

| Sub-object | OBJECT ID | Notes |
|---|---|---|
| Title row | `execution-log-entry-{id}-title` | Same as 01.7 |
| **Status badge — modification variant** | `execution-log-entry-{id}-status-mod` | "EXECUTED W/ MOD" — amber/severity-warning fill; visually distinct from "EXECUTED" |
| Action route + timestamp | `execution-log-entry-{id}-route` | Same as 01.7 |
| IdP identity | `execution-log-entry-{id}-identity` | Same as 01.7 |
| **Reason preview row** | `execution-log-entry-{id}-reason-preview` | First ~80 chars of the reason text, ellipsis if longer; click → expand drawer |
| **View original affordance** | `execution-log-entry-{id}-view-original` | Small link "Show original" — keeps Stefan from having to expand the full drawer |
| Outcome status row | `execution-log-entry-{id}-outcome` | Same as 01.7 — pending → confirmed/unverified/divergent |
| Expand chevron | `execution-log-entry-{id}-expand` | Opens detail drawer with full diff view |

### Drawer Content (modification variant)

When Stefan (or later, Renate, or the auditor) expands the log entry, the drawer shows:

1. **Header:** RAO id + modification timestamp + Stefan's IdP identity
2. **Original payload** (read-only, formatted same as 01.5 RAO card ACTION section)
3. **Modified payload** (read-only, fields that changed highlighted)
4. **Diff view** (clear visual of what changed; same component as 02.4 diff summary)
5. **Reason for modification** (full text)
6. **Original WHY + STEPS** (the justification Stefan saw before modifying)
7. **Regenerated WHY + STEPS** (the justification reflecting the modification — for downstream readers)
8. **Outcome chain** (hold gesture timing, atomic write, HAFAS poll history, outcome status)

This is the row Renate's audit committee will read when verifying that modifications are honestly captured.

## Technical Notes

- **Modification row schema:** the audit row stores `original_rao_id` (joins to the original RAO row) plus the modification payload, reason, and the outcomes of the *modified* action. Renate's drawer view joins both for display.
- **Modification rate computation:** at the pilot evidence dashboard (scenario 08), modification rate = `modified_executions / total_executions`. Captured separately from acceptance rate; the two together form the composite signal Phase 2 trust composite watches.
- **Badge colour rationale:** "EXECUTED W/ MOD" uses amber/warning treatment (not red/critical) — it's *not a problem*; it's *captured nuance*. Amber communicates "additional context, worth a look" without alarm.
- **Drawer accessibility:** the diff view must be navigable via screen reader — both original and modified payloads should be announced with clear labels.
- **Print/export friendliness:** Renate's audit committee may export the drawer view as PDF; ensure print stylesheet renders both payloads side-by-side with diff highlights preserved.
