## Overview

**Page Purpose:** Same RAO card component as 01.5, **rendered inline in the expanded triage card** (instead of in a chat conversation). The grammar, fields, and operator-verb buttons are identical. The only difference is the container — and the path Stefan takes from here (Modify is the primary path in this scenario; Mark executed in 01.5).

**User Situation:** Stefan clicked "Generate recommendation" in 02.2. The RAO has streamed in. He reads payload + WHY + STEPS + confidence pill. He decides he wants to modify the maintenance slot.

**Success Criteria:**
- Identical card grammar to 01.5 — same payload structure, same operator-verb buttons
- RAO appears in the expanded card region (replacing the source data hyperlink view)
- "Modify" is given equal visual weight to "Mark executed" (modification rate is a signal in its own right per Stefan persona detail)

**Entry Points:**
- "Generate recommendation" from [02.2-card-expand](../02.2-card-expand/02.2-card-expand.md)

**Exit Points:**
- "Modify" → [02.4-modify-panel](../02.4-modify-panel/02.4-modify-panel.md)

## Inheritance

**This page inherits 100% of structural specification from [01.5-rao-card](../../01-stefan-crisis-redeployment/01.5-rao-card/01.5-rao-card.md).** Layout, spacing, typography, sections, all object IDs use the `rao-card-*` namespace.

The differences below are *contextual*, not structural.

### Contextual Differences from 01.5

| Aspect | 01.5 (Scenario 01) | 02.3 (Scenario 02) |
|---|---|---|
| Container | Chat message bubble in crisis full-width (01.3) | Inline region within expanded triage card (02.2) |
| Entry trigger | Stefan free-form query in chat | Stefan clicked "Generate recommendation" button |
| Primary exit (most-likely path) | Mark executed | Modify (this scenario captures the modification flow) |
| Visual prominence of buttons | "Mark executed" primary, "Modify" secondary | Both buttons same weight (because modification IS the scenario) |
| Surrounding context | Conversation thread with prior tool calls | Cost-impact + fault detail visible above; no chat thread |

### Button Visual Weight (the one structural delta)

In 02.3, "Modify" and "Mark executed" both carry **equal prominence** — same size, same touch target, same vertical alignment — to communicate that modification is a first-class operator response, not a second-tier choice.

**Winston's refinement (party mode 2026-05-25):** equal weight ≠ identical fill. To avoid "indecisive" reading, Modify renders as outlined/secondary-fill, Mark Executed as solid primary fill, but both with identical font weight, height, and horizontal weight. Reads as "two real choices, both first-class," not "the system couldn't pick."

**Mimir's build note (party mode):** both buttons MUST carry distinct `data-action` attributes (`data-action="modify"` / `data-action="mark-executed"`) so analytics can disambiguate modification rate from acceptance rate — this signal is required regardless of visual treatment.

The verb rule is unchanged. Both verbs identify the operator as actor.

| Button | Style in 01.5 | Style in 02.3 | data-action |
|---|---|---|---|
| "Modify" | Secondary (outline) | **Outlined/secondary-fill, primary prominence** | `modify` |
| "Mark executed" | Primary (filled) | Solid primary fill, primary prominence | `mark-executed` |

The buttons are equal-prominence; the order is "Modify" first (left), "Mark executed" second (right), reflecting the most-likely path in this scenario but not enforcing it.

**On LOW-confidence RAOs in scenario 02:** same DISABLE rule as 01.5 applies. "Mark executed" is greyed out; only "Modify" is interactive. (See [01.5-rao-card](../../01-stefan-crisis-redeployment/01.5-rao-card/01.5-rao-card.md) Low confidence state.)
